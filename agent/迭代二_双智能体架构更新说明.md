# 迭代二更新说明：双智能体解耦架构 (Double Agent Pipeline)

## 1. 背景与痛点
在迭代二初期的学术调研 Agent 开发中，我们遇到了大语言模型（LLM）由于 ReAct 架构限制导致的内容截断、反复输出占位符（如“本章将深入探讨”等空话）以及上下文丢失等问题。

问题的根本原因在于，原本单一的 Agent 需要同时兼顾：
1. **工具调用与逻辑规划**（严格的 JSON 格式要求，如调用 arXiv 检索和文件读写）。
2. **长文本创作**（耗费巨大的注意力去构思和输出数千字的长文）。

这两种心智模式在一定程度上是互相排斥的，把数千字的 Markdown 塞进 JSON Value 里也极易导致解析错误（SyntaxError/DecodeError），从而触发大模型的“保护性偷懒”机制。

## 2. 架构演进：双智能体解耦 (Double Agent Pipeline)
为了打破长文本生成的限制，我们对 gent/main.py 的核心流程进行了重构，拆分出了职能明确的 **Researcher Agent** 与 **Writer Agent**。

### 2.1 阶段一：Researcher Agent (纯粹的学术调研员)
*   **职责**：专注于搜索、阅读和外围知识的抓取。
*   **工作流**：
    1. 使用 clear_notes 初始化外部记忆草稿本。
    2. 使用 rxiv_search 精确检索领域内的前沿核心论文。
    3. 使用 rxiv_pdf_reader 深入读取 PDF 正文原文。
    4. 一旦抓取到关键信息，立即使用 ppend_note 将重要公式、网络架构、实验数据写入外部草稿本（esearch_notes.txt）。
*   **核心变化**：仍然保留完整的 ReAct 循环，但严格禁止其输出总结性长文。剥离写作负担，写完笔记（史诗级摘要）后即刻结束任务。

### 2.2 阶段二：Writer Agent (无束缚的顶级撰稿人)
*   **职责**：专注于纯文本的深度加工与长篇学术撰稿。
*   **工作流**：
    1. 完全脱离 ReAct 束缚，不再提供任何 Tool，也不要求输出 JSON。
    2. 直接读取前一阶段产出的 esearch_notes.txt 外部记忆库。
    3. 采用直接对话（Direct Prompting）模式，将数百字的硬核笔记原汁原味地发散为包含背景、深度技术解析与未来展望的数千字超长篇连贯综述。

## 3. 架构重构对团队协作的指导意义
本次更新为我们六人软工团队提供了一个非常关键的设计理念参考——**好钢用在刀刃上，并非所有的过程都要生硬地套入 Agent 工具循环**。

*   **Tools/RAG 研发视角**：可以更专注地为 Researcher 扩充数据源库（如引入 Google Scholar、Semantic Scholar或本地知识库），不用再担心 Prompt 太长导致模型写不出文章。
*   **Frontend/UI 视角**：可以分别对两个阶段进行差异化的状态可视化展示：阶段一显示清晰的日志跟踪与加载条（Search -> Read -> Note），阶段二显示流式的 Markdown 渲染打字机效果。
*   **Evaluation/测试视角**：现在能够使用 Ragas 框架更精准地评估不同模块的短板：
    *   针对 **Researcher** 评估“信息查全率 / 检索准确度”。
    *   针对 **Writer** 评估“文本连贯性 / 知识无幻觉度 / 学术深度”。
