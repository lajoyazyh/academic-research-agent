# 版本更新说明 (相对于 commit: 8267ed795c74ae6635994ce8fc05d63dcbca9132)

本次更新主要集中在三个方面：**彻底解决部分模型（智谱 GLM-4 等）在 Ragas 评估时的 API 调用异常问题**、**优化大模型评测阶段的上下文噪音清洗**，以及**引导 Agent 产生更有深度的推理思考**。

## 1. 底层 API 兼容性结构重构（修复 `400 - 1210` 参数错误）
由于 LangChain / Ragas 会在后台隐式注入多种 OpenAI 独有或者非标准的请求参数，导致在对接智谱等存在严格参数校验的 API 时会连续崩溃。

- **全面弃用黑名单机制，启用极严苛的参数白名单**：在 `_ZhipuCompatibleChatOpenAI` 中，对原生底层请求客户端（`self.client.create` 与 `self.async_client.create`）使用强效的 Monkey Patch 进行接管拦截。
  - **顶级参数过滤**：仅允许 `model`, `messages`, `temperature`, `top_p` 等必要合法键值通过，无视并强制剥离 `seed`, `response_format`, `n`, `logprobs` 等导致报错的隐藏字段。
  - **消息内部清洗**：在 `messages` 数组中，限制每个字典只准保留 `{"role", "content", "tool_calls", "tool_call_id"}`。彻底杜绝 Langchain 模拟用户时私自产生的 `name` 等非法属性导致验证失败。
- **浮点阈值边界修复**：针对 Ragas 等工具经常传入极小低随机性（如使得 temperature 转化为科学计数法 `1e-08` 或者 `0.0`），导致智谱 API 后端抛出 1210 格式错误的经典 Bug，统一在拦截器内设置了 `temperature` 最小值为 `0.01`，`top_p` 最大值为 `0.99`。
- **排错机制升级**：现在一旦底层 API 调用抛出异常，触发拦截器会自动抓取序列化的全部负载打印保存至本地的 `zhipu_error_payload.json`，极大提升了模型接口排错反馈链路。

## 2. 评估管线与指标计算体系（Metrics）的调整
此前在使用 Ragas 跑诸如基于指令生成的评测大模型时，部分指标出现了极端的失效情况。

### 为什么移除 `answer_relevancy`？
Ragas 的 `answer_relevancy` 指标基于一个**关键假设**：存在一份"标准金标准答案"（Ground Truth），指标通过让大模型比较生成答案与金标准答案的语义相似度来判定答案的相关性。但在我们的使用场景中：
- **我们没有预定的标准答案**，评测的是 Agent 遵循指令完成任务的能力（例如"检索与某主题相关的论文并汇总"）。
- 当 Ragas 无法生成或找到金标准答案时，它会采用**反向工程**的方式，通过分析给定的上下文和原始问题来"推断"一个答案。但对于指令导向的任务，这种反向推断会失败或产生极低的匹配分数（通常为 0 分）。
- 因此 `answer_relevancy` 在我们的评测框架中**本质上不适用**，会持续产生无效数据。

**解决方案**：用自研的 LLM-as-a-judge 方案中的 `relevance_score` 替代，由大模型直接判断答案是否符合题意和指令要求，规避了对"金标准"的依赖，更贴合指令完成度评估的逻辑。

### 其他指标调整
- **增加推理评分项**：在综合评价大类中，补充并加权计算 `reasoning_score`，使过程导向更看重 Agent 思考连贯性。
- **更新 Judge 大模型焦点提示词**：同步修订了内部大模型裁判的侧重点定义（`judge_focus`），额外突出了“指令相关性”的考察比重。

## 3. Context 噪音清洗 (`_clean_context_string`)
为了解决由于系统长日志污染引发的 `context_precision` 评估拉跨的问题，做出了显式的 Context 清理。
- 注入了专用的上下文清洗器，专门过滤在 Agent 调用文档和工具环境时携带的长尾警告（如 `...[警告：为防止大模型上下文超限，PDF后续内容已被系统自动截断]...`）以及任务汇报类长文本噪音（如 `✅ 笔记已成功...`）。

## 4. Agent 底层 Prompt 结构引导深度推理优化
- **修改了系统级指导说明 `build_system_prompt`**：直接通过系统定义对齐强制约束。在 ReAct 循环内的 `thought` 推理过程要求中，不再进行宽泛解释，而是**强制其展示深度的检索推理（Reasoning）逻辑**：需要明确说明选择特定文档论文的原因、比较各种关联搜索互补性，并显式反思是否足够全面。这有利于评估后续打出更好的 `reasoning_score`。

## 5. 项目环境与工程化清理
- 更新了 `.gitignore`，将通配符的忽略范围由单一定向排除扩展至 `*.db`，隔离了测试或生产产生的临时本地数据库文件被污染进入 Git 追踪（如 `eval_platform.db` 被剥离仓库跟踪并删除）。
