# 迭代三 Agent 优化文档

**版本**：v4.0
**日期**：2026-06-18
**范围**：迭代三 Academic Research Agent 的会话化、交互式调研、RAG、Skills、深度分析、跨 Session 知识共享、评估与 CI/CD 支撑
**依据**：迭代三任务说明、项目当前实现、`evaluation/` 评估材料、2026-05 至 2026-06 git 提交记录、阶段性评估结果

---

## 1. 概述

迭代三的目标不是简单给迭代二 Agent 增加几个页面，而是把原来的“一次性自动生成器”改造成一个可持续工作的学术调研系统。用户不再只是输入主题并等待最终综述，而是可以在关键词、论文筛选、笔记、分析、综述、修订等关键节点介入判断；系统也不再只保留最终结果，而是把计划、搜索轨迹、论文、笔记、分析、草稿、对话历史和评估材料组织成可复盘的 Session。

最终系统形成了两条等价的使用路径：

- **手动分步模式**：用户逐步确认关键词、检索论文、选择论文、生成和编辑笔记、生成和编辑分析卡片、生成和修订综述。
- **一键自动模式**：系统按照 `规划 -> 搜索 -> 笔记 -> 分析 -> 综述` 自动执行完整 pipeline，同时保留中间产物和 trace，便于后续检查和修改。

迭代三的核心改造可以概括为六波：

```text
初始状态：迭代二一次性 Agent
  -> 第一波：Session 状态管理 + 多轮聊天
  -> 第二波：深度笔记生成 + 聊天增强 + 工具注册中心
  -> 第三波：RAG 检索系统升级
  -> 第四波：Pipeline 增强 + Skills + Paper Register
  -> 第五波：Web 模块化 + 上下文窗口管理
  -> 第六波：分析阶段并入主流程 + Skill trace + Copilot 范围选择
```

## 2. 迭代二痛点与迭代三目标

### 2.1 迭代二主要问题

| 问题 | 表现 | 对用户/答辩的影响 |
|------|------|------|
| 流程黑盒 | 用户只能看到最终综述，无法看到每步判断 | 难以解释 Agent 为什么选择某些论文或得出某些结论 |
| 无 Session 状态 | 每次运行都是一次性目录，难以恢复和追加 | 无法支持连续调研、断点继续、多轮修改 |
| 中间产物不可控 | 关键词、论文、笔记无法稳定人工介入 | 综述质量不可控，错误难以及时纠正 |
| 聊天能力弱 | 单轮问答或简单上下文拼接 | 无法自然追问，也难以让 Agent 根据反馈修订 |
| RAG 能力有限 | 仅 BM25 或粗粒度文本截断 | 对同义表达、跨语言问题、论文全文利用不足 |
| 工具管理分散 | 工具硬编码，启用状态不可视 | 演示时难以控制不稳定外部 API |
| Prompt 不可定制 | 不同主题只能使用同一套默认策略 | 缺少面向不同领域和用户偏好的可控性 |
| 评估与可证明性弱 | 缺少 trace、Skill 状态、评估脚本闭环 | 难以证明系统确实按设计运行 |

### 2.2 迭代三设计目标

1. **交互式调研**：允许用户参与关键词、论文、笔记、分析和综述决策。
2. **连续对话**：一个研究主题可长期维护，支持多会话聊天、历史保存、上下文压缩。
3. **自动与手动统一**：手动分步和一键自动使用同一套 pipeline，避免两套逻辑不一致。
4. **深度 RAG**：基于论文全文/摘要生成结构化笔记，并为问答和综述提供可靠上下文。
5. **可观测和可复盘**：记录 ReAct trace、错误类型、Skill 状态、运行阶段和中间产物。
6. **可评估**：提供 `evaluation/` 独立评估入口、测试脚本和 fallback 机制。

## 3. 当前正式 Pipeline

### 3.1 阶段流转

```text
创建 Session
  -> 规划：生成关键词与检索计划
  -> 搜索：ReAct Agent 调用学术工具检索论文
  -> 笔记：RAGNoteGenerator 基于论文全文/摘要生成结构化笔记
  -> 分析：生成文献对比、研究脉络、研究空白
  -> 综述：将笔记和分析洞察注入 Writer，生成综述草稿
  -> 修订：用户通过 Markdown 编辑器或对话反馈继续修改
```

### 3.2 分步模式

分步模式强调用户参与关键判断：

- 关键词阶段：AI 生成初始关键词，用户可编辑确认。
- 论文阶段：用户可检索、添加 arXiv ID/链接、上传 PDF、删除论文、调整论文状态。
- 笔记阶段：用户选择论文生成 RAG 笔记，并可在 Markdown 面板中修改。
- 分析阶段：系统生成三张分析卡片，用户可分别编辑。
- 综述阶段：系统基于笔记和分析写作，用户可继续对话修订。

### 3.3 自动模式

自动模式解决快速端到端执行需求：

```text
规划 -> 搜索 -> 笔记 -> 分析 -> 综述
```

自动模式不是另一套简化逻辑，而是对同一 pipeline 的自动调度。它会在后台线程中执行，前端通过轮询展示阶段、进度、trace 和最终结果。用户可在搜索过程中停止运行，结束后仍可回到分步界面继续编辑。

### 3.4 分析阶段并入主流程

迭代三后期将分析阶段从“展示性页面”升级为正式 pipeline 节点：

- 自动模式中，分析阶段位于笔记之后、综述之前。
- 分析结果保存到 `sessions/{session_id}/analysis/analysis_results.json`。
- 前端展示为三张可编辑 Markdown 卡片：`compare`、`lineage`、`gaps`。
- Writer 读取分析结果，将其作为额外写作上下文。
- 若分析结果不存在，Writer 自动回退到旧流程，仅基于笔记生成综述。

这使分析结果真正参与综述写作，而不是孤立的页面输出。

## 4. 第一波：Session 状态管理与多轮聊天

### 4.1 问题

迭代二每次运行创建独立结果目录，缺少统一状态和长期记忆：

- 运行中无法暂停，也不能从搜索、笔记、写作等阶段恢复。
- 搜索结束后若发现关键词有问题，只能重跑整个流程。
- 每次运行数据孤立，论文、笔记、草稿和聊天无法统一绑定。
- 用户追问“它和前面那篇相比如何”时，系统无法可靠理解指代。

### 4.2 Session 状态机

迭代三引入 `SessionManager`，将一个研究任务抽象为一个 Session。状态机覆盖：

```text
planning -> plan_confirmed -> searching -> search_complete
-> reviewing_notes -> writing -> reviewing_draft -> complete
```

关键设计是允许多个稳定状态回到 `searching`，从而支持追加调研：

```text
search_complete -> searching
reviewing_notes -> searching
writing -> searching
reviewing_draft -> searching
complete -> searching
```

这使“发现论文不够，再补充搜索”成为正常流程，而不是异常操作。

### 4.3 Session 目录结构

```text
sessions/{session_id}/
├── metadata.json
├── plan/
│   ├── initial_plan.md
│   └── confirmed_keywords.json
├── papers/
│   ├── papers_list.json
│   ├── deleted_papers.json
│   ├── *.pdf
│   └── *.txt
├── notes/
│   ├── draft_notes.md
│   └── edit_history/
├── analysis/
│   ├── analysis_results.json
│   ├── compare.md
│   ├── lineage.md
│   └── gaps.md
├── draft/
│   ├── draft_v*.md
│   ├── current_draft.md
│   └── user_feedback.md
├── traces/
│   └── run_traces.json
└── chats/
    ├── _index.json
    └── conv_*.json
```

### 4.4 自动修复和数据标准化

Session 加载时会检测卡住状态。例如状态长时间停在 `searching`、`writing` 等运行中状态，但已有稳定产物时，系统会自动回退或修复到对应稳定状态，避免演示时卡死。

论文数据也会被标准化：

- `papers_list.json` 统一保存标题、作者、摘要、来源、状态、PDF 路径等字段。
- `deleted_papers.json` 记录用户删除过的论文，避免重新搜索时反复出现。
- 支持恢复误删论文。

### 4.5 多轮聊天

每个 Session 支持多个独立聊天会话：

```text
chats/
├── _index.json
└── conv_{uuid}.json
```

聊天系统会读取最近若干轮对话、当前论文、笔记、草稿和 RAG 检索结果，使用户能够自然追问、比较论文，并要求 Agent 修改笔记或综述。

### 4.6 第一波效果

| 维度 | 优化前 | 优化后 |
|------|--------|--------|
| 会话持续性 | 一次性运行 | Session 状态机 + 目录持久化 |
| 断点继续 | 不支持 | 支持从稳定阶段恢复 |
| 追加调研 | 不支持 | 多状态可回到搜索 |
| 对话记忆 | 单轮或弱上下文 | 多会话历史保存 |
| 数据组织 | 运行结果分散 | 论文、笔记、草稿、聊天统一绑定 |

## 5. 第二波：深度笔记生成、聊天增强与工具注册中心

### 5.1 深度笔记生成

#### 问题

早期笔记生成主要依赖自由格式输出和粗粒度文本输入，存在：

- 笔记结构不统一，后续综述难以稳定引用。
- 只读取 PDF 前几页时，可能漏掉实验、消融、局限等关键内容。
- PDF 提取失败或 Embedding 失败时，流程容易中断。
- 用户无法为不同研究主题定制笔记关注点。

#### 实现

`RAGNoteGenerator` 将笔记拆成多个固定维度：

| 章节 | 关注点 |
|------|------|
| 研究背景 | 问题、挑战、动机、已有方法局限 |
| 核心方法 | 模型、算法、框架、关键创新 |
| 实验设置 | 数据集、基线、指标、实验条件 |
| 关键结果 | 性能、对比、主要发现 |
| 消融与分析 | 组件贡献、案例分析、可视化 |
| 亮点与不足 | 贡献、局限、未来工作 |

生成流程：

```text
PDF/摘要文本
  -> 分块
  -> Embedding 或 BM25 检索相关片段
  -> 分章节生成结构化笔记
  -> 自修复检查
  -> 保存 Markdown
```

降级策略：

- PDF 提取失败：回退到摘要生成。
- Embedding API 失败：回退到 BM25。
- LLM 生成失败：保留模板化基础笔记。

### 5.2 聊天系统增强

聊天系统不再只是把所有内容粗暴拼接，而是引入上下文优先级：

```text
用户当前消息 > RAG 检索结果 > 对话历史 > 笔记/草稿 > 摘要
```

同时加入意图识别：

- 普通问答：直接回答用户问题。
- 修订请求：识别目标是笔记、分析还是综述。
- 高置信度修订：请求用户确认后执行。
- 低置信度：要求用户澄清。

为了降低延迟，系统先用规则识别明显的“修改/删除/补充/重写”等指令，只有无法判断时才调用 LLM 进行意图分类。

### 5.3 工具注册中心

迭代三将工具集中到 `core/tool_registry.py` 管理。工具元数据包括：

- 名称
- 描述
- 类别
- 所属 pipeline 阶段
- 启用状态
- 参数配置

工具配置持久化到 `config/tools.json`，前端工具管理面板可进行启用、禁用、批量操作和恢复默认。

当前工具按类别包括：

| 类别 | 工具示例 | 用途 |
|------|------|------|
| 学术搜索 | `arxiv_search`、`openalex_search`、`crossref_search` | 搜索论文 |
| 论文获取 | `arxiv_fetch`、`crossref_fetch_doi` | 获取元数据 |
| PDF | `arxiv_pdf_reader`、`arxiv_download_pdf` | 阅读和下载 PDF |
| 文件操作 | `clear_notes`、`append_note` | 记录研究笔记 |
| 笔记生成 | `rag_note_generator` | RAG 深度笔记 |
| 收录管理 | `paper_register` | 论文登记和 PDF 下载 |
| 对话检索 | `retriever` | Session 内问答检索 |

部分外部 API 不稳定或容易限流，因此默认禁用。这是演示稳定性策略，不是核心功能缺失。

### 5.4 第二波效果

| 维度 | 优化前 | 优化后 |
|------|--------|--------|
| 笔记结构 | 自由格式 | 多章节结构化 |
| 原文利用 | 粗粒度截断 | 全文分块检索 |
| 异常处理 | 失败即中断 | PDF/Embedding/LLM 多层 fallback |
| 工具管理 | 分散硬编码 | 注册中心 + 前端开关 |
| 聊天修订 | 难以区分问答和修改 | 意图识别 + 确认式修订 |

## 6. 第三波：RAG 检索系统升级

### 6.1 问题

仅依赖 BM25 或关键词匹配时，系统容易漏掉语义相近但字面不同的内容。例如用户问“训练稳定性”，论文中可能写的是 “training instability” 或 “optimization robustness”。中文问题检索英文论文也会受到影响。

### 6.2 混合检索

`HybridRetriever` 同时使用：

- **BM25/TF-IDF 通路**：保证关键词精确匹配。
- **Embedding 通路**：捕捉语义相似度。

综合得分采用加权融合：

```text
score = 0.3 * BM25_score + 0.7 * embedding_score
```

该权重让 Embedding 主导语义理解，同时保留 BM25 对专有名词、模型名、数据集名的精确召回。

### 6.3 缓存与降级

为了降低重复向量化成本，系统会将 Embedding 缓存在磁盘中。论文更新后会失效对应缓存。

降级路径：

```text
Embedding 可用 -> 混合检索
Embedding 失败 -> 纯 BM25
BM25 构建异常 -> 降低特征数后重试
检索结果不足 -> 迭代式扩展查询
```

### 6.4 迭代式检索

当第一轮检索结果数量不足或最高分过低时，系统会生成扩展查询，从不同角度补充检索，再合并去重排序。这样能减少一次查询表达不准导致的召回失败。

### 6.5 第三波效果

| 维度 | 优化前 | 优化后 |
|------|--------|--------|
| 检索方式 | 单一路径 | BM25 + Embedding |
| 语义理解 | 弱 | 支持同义表达和跨语言表达 |
| 缓存 | 每次重算 | Embedding 磁盘缓存 |
| 检索失败 | 无补救 | 迭代式扩展查询 |
| 异常处理 | 易中断 | 纯 BM25 回退 |

## 7. 第四波：Pipeline 增强、Skills 与 Paper Register

### 7.1 问题

前三波解决了会话、笔记和检索问题，但 pipeline 仍有若干工程风险：

- Prompt 固定，无法针对不同学科、不同写作目标定制。
- Writer 有时不严格遵守结构，章节重复或遗漏。
- 搜索阶段下载 PDF 和登记论文是两个动作，可能出现数据不一致。
- Agent 忘记记录笔记时，Writer 阶段缺少输入。
- 搜索过程阻塞，用户无法中断。

### 7.2 Skills 双通道注入

Skills 支持三个阶段：

| 阶段 | Skill 类型 | 作用 |
|------|------|------|
| 搜索 | `search` | 定义搜索策略、数据源偏好、筛选标准 |
| 笔记 | `notes` | 定义笔记结构、关注维度、摘要方式 |
| 综述 | `write` | 定义综述结构、写作风格、引用方式 |

双通道机制：

```text
阶段入口
  -> Skill 存在且有效：注入 Skill 策略
  -> Skill 缺失/为空/加载失败：回退默认策略
```

这样既支持用户定制，又保证系统不会因 Skill 配置错误而崩溃。

### 7.3 综述自修复

Writer 生成后进入轻量检查：

1. 检查是否遵守目标结构。
2. 检查是否有明显重复、空章节或占位符。
3. 必要时进行二次修复。

如果修复失败，则保留原始草稿，避免“修复”反而破坏可用结果。

### 7.4 轨迹兜底笔记

若 Agent 搜索后没有形成完整笔记，系统会扫描 traces 中的 `arxiv_search`、`arxiv_fetch`、`crossref_search` 等 observation，提取候选论文信息，生成兜底笔记。这保证 Writer 不会因为笔记缺失完全无法执行。

### 7.5 Paper Register

`paper_register` 将“论文登记 + PDF 下载”合并为单次动作：

```text
发现论文 -> 下载 PDF -> 提取文本 -> 写入 papers_list.json
```

这样减少“下载了但没登记”或“登记了但没有 PDF”的不一致。

### 7.6 后台执行与取消

搜索和自动模式通过后台线程运行，前端轮询当前状态。用户可在搜索过程中点击停止，避免长时间外部 API 调用阻塞页面。

### 7.7 第四波效果

| 维度 | 优化前 | 优化后 |
|------|--------|--------|
| Prompt 定制 | 硬编码 | search/notes/write Skills |
| 写作质量 | 一次生成 | 结构检查和自修复 |
| 笔记缺失 | 流程失败 | trace 兜底笔记 |
| 数据一致性 | 下载/登记分离 | Paper Register 一体化 |
| 执行体验 | 同步阻塞 | 后台线程 + 可取消 |

## 8. 第五波：Web 模块化与上下文窗口管理

### 8.1 问题

随着功能增加，早期 `web_app.py` 承担了应用初始化、页面路由、API、模型定义、聊天逻辑、Agent 调度等大量职责，文件膨胀到难以维护的程度。另一方面，聊天上下文越来越长，需要可视化和压缩机制。

### 8.2 Web 应用解耦

后端拆分为：

```text
web_app.py
backend/routes/
├── deps.py
├── models.py
├── pages.py
├── session.py
├── chat.py
├── conversation.py
├── draft.py
├── agent.py
└── admin.py
```

设计要点：

- `web_app.py` 只负责 FastAPI 初始化、静态资源和路由注册。
- `models.py` 统一管理 Pydantic 请求/响应模型。
- `deps.py` 延迟注入共享依赖，减少循环导入。
- 路由按功能聚合，便于测试和定位问题。

### 8.3 上下文窗口管理

新增上下文统计和压缩能力：

- `GET /api/sessions/{id}/context/stats`：返回消息数、轮次、估算 token、使用比例。
- `POST /api/sessions/{id}/context/compress`：将早期对话摘要为较短上下文，保留最近几轮原文。

前端聊天区显示 token 使用进度条，当对话变长时提示用户压缩。

### 8.4 前端工作台升级

前端从简单页面演进为研究工作台：

- 首页：项目概览、最近活动、收藏夹、全局 Copilot。
- 控制台：关键词、论文、笔记、分析、综述、轨迹、工具等标签页。
- Markdown 编辑：笔记、分析卡片、综述均支持 Vditor 编辑/渲染。
- 轨迹区：展示 thought/action/observation、Skill 状态、错误信息。
- 自动模式：统一进度展示和完成后切换到综述视图。

### 8.5 第五波效果

| 维度 | 优化前 | 优化后 |
|------|--------|--------|
| 后端结构 | 单文件膨胀 | 路由模块化 |
| 依赖管理 | 容易循环导入 | `deps.py` 延迟注入 |
| 上下文长度 | 隐式截断 | 可视化统计 + 压缩 |
| 前端体验 | 结果展示 | 工作台式交互 |
| Markdown 编辑 | 局部支持 | 笔记/分析/综述统一支持 |

## 9. 第六波：分析注入、Skill Trace 与 Copilot 范围选择

### 9.1 分析结果接入综述写作

分析阶段生成三类内容：

| 分析类型 | 写作价值 |
|------|------|
| `compare` | 支撑方法对比、实验对比、优缺点比较 |
| `lineage` | 支撑技术演进、研究脉络、范式变化 |
| `gaps` | 支撑研究空白、未来方向和开放问题 |

写作阶段读取 `analysis/analysis_results.json`。如果存在分析内容，则将其作为“分析洞察上下文”注入 Writer prompt；如果不存在，保持原流程兼容。

这一改动使系统从“先做笔记再写综述”升级为“先做笔记，再综合分析，再写综述”。

### 9.2 分析卡片可编辑

分析界面采用与笔记、综述一致的 Markdown 编辑体验。三张卡片分别可编辑保存，避免原先直接显示未渲染 Markdown 文本的问题。

用户可以在综述生成前修改分析结果，因此 Writer 使用的是用户审核后的分析洞察，而不只是模型初稿。

### 9.3 Skill 行为可证明

为了在答辩时证明 Skill 确实进入 Agent 流程，系统在 search / notes / write 三个阶段记录 `SKILL_STATUS` trace：

```json
{
  "event": "SKILL_STATUS",
  "stage": "write",
  "skill_id": "custom_write_skill",
  "skill_title": "custom_write_skill",
  "loaded": true,
  "fallback": false
}
```

记录内容至少包括：

- 阶段
- skill_id
- skill_title
- 是否成功加载
- 是否回退默认策略
- 回退原因

### 9.4 跨 Session 知识共享显式选择

全局 Copilot 原先默认从所有真实 Session 中检索。迭代三后期增加范围选择：

- `/api/knowledge/chat` 支持可选 `session_ids`。
- 不传 `session_ids`：保持全局检索。
- 传入 `session_ids`：只在选中 Session 的 chunks 中检索。
- 前端 Copilot 区域增加 Session 多选入口，默认选择全部真实 Session。

这使“跨对话知识共享”从隐式全局能力变成用户可控制的知识范围。

### 9.5 第六波效果

| 维度 | 优化前 | 优化后 |
|------|--------|--------|
| 分析阶段 | 页面展示为主 | 正式进入 pipeline |
| 综述写作 | 主要依赖笔记 | 笔记 + 分析洞察 |
| Skill 证明 | 只能看配置 | trace 中显示 active/fallback |
| Copilot 范围 | 默认全局 | 可选一个或多个 Session |
| 答辩展示 | 依赖口头解释 | 可用页面 trace 和文件产物证明 |

## 10. 评估结果与解释

### 10.1 评估指标

| 指标 | 含义 |
|------|------|
| context_precision | 检索上下文中相关信息的精确度 |
| context_recall | 检索上下文覆盖相关信息的程度 |
| correctness_score | 回答或综述中事实与结论的正确性 |
| completeness_score | 内容覆盖任务要求和关键信息的完整程度 |
| grounding_score | 生成内容与可检索资料、论文证据之间的支撑程度 |
| step_coherence_score | Agent 执行步骤、工具调用和中间推理的连贯程度 |
| faithfulness_score | 生成内容对资料的忠实程度 |
| helpfulness_score | 回答或综述对用户的帮助程度 |
| relevance_score | 内容与主题或问题的相关性 |
| reasoning_score | 推理质量、结构化程度和逻辑连贯性 |
| overall_score | 综合评分 |

### 10.2 阶段评估数据

评估部分依据最新五轮实验数据整理。五轮数据对应从基础版本到迭代三增强版本的逐步优化过程：

| 指标 | 第一轮 | 第二轮 | 第三轮 | 第四轮 | 第五轮 |
|------|------:|------:|------:|------:|------:|
| context_precision | 0.303 | 0.374 | 0.450 | 0.476 | 0.571 |
| context_recall | 0.75 | 0.75 | 0.83 | 0.87 | 0.90 |
| correctness_score | 0.75 | 0.80 | 0.81 | 0.85 | 0.90 |
| completeness_score | 0.75 | 0.75 | 0.82 | 0.84 | 0.88 |
| relevance_score | 0.75 | 0.85 | 0.86 | 0.88 | 0.94 |
| grounding_score | 0.65 | 0.70 | 0.80 | 0.81 | 0.83 |
| reasoning_score | 0.65 | 0.65 | 0.78 | 0.86 | 0.80 |
| step_coherence_score | 0.75 | 0.80 | 0.85 | 0.90 | 0.88 |
| faithfulness_score | 0.75 | 0.75 | 0.81 | 0.81 | 0.92 |
| helpfulness_score | 0.75 | 0.70 | 0.76 | 0.85 | 0.83 |
| **overall_score** | **0.65** | **0.70** | **0.80** | **0.80** | **0.83** |

### 10.3 结果分析

- `overall_score` 从第一轮 0.65 提升到第五轮 0.83，说明迭代三优化不是只增加界面功能，而是带来了可量化的整体质量提升。
- `context_precision` 从 0.303 提升到 0.571，累计提升 0.268，主要受益于 Session 化上下文组织、RAG 检索升级、论文/笔记/分析产物的结构化沉淀。
- `context_recall` 从 0.75 提升到 0.90，说明系统在引入全文检索、Embedding/BM25 fallback 和分析阶段后，对相关资料的覆盖能力更强。
- `correctness_score`、`completeness_score`、`relevance_score` 均持续提升，说明生成内容在事实正确性、信息完整度和主题贴合度上都有稳定改善。
- `grounding_score` 从 0.65 提升到 0.83，表明综述和问答越来越依赖可追溯的论文、笔记和分析上下文，而不是泛泛生成。
- `reasoning_score` 在第四轮达到 0.86，第五轮为 0.80，说明推理能力整体显著强于初始版本，但最终版本在加入更多约束和可控机制后，推理深度与稳定性之间仍存在权衡。
- `step_coherence_score` 从 0.75 提升到 0.88，说明 Agent 的阶段衔接、工具调用和中间步骤更加连贯，自动模式和 trace 机制对流程稳定性有正向作用。
- `faithfulness_score` 在第五轮达到 0.92，是提升最明显的指标之一，说明分析注入、结构化笔记和写作自修复增强了输出对资料的忠实程度。
- `helpfulness_score` 第二轮短暂下降到 0.70，随后提升到 0.83，说明早期会话化和上下文扩展可能带来一定噪声，但后续 RAG、分析和写作流程优化弥补了这个问题。

总体来看，五轮评估呈现明确上升趋势。第五轮相对第一轮的主要增幅包括：`overall_score` +0.18、`context_precision` +0.268、`context_recall` +0.15、`faithfulness_score` +0.17、`relevance_score` +0.19。这说明迭代三优化在检索、证据支撑、流程连贯性和最终生成质量上均取得了实质性改善。

### 10.4 evaluation 目录

## 14. 设计原则总结

### 14.1 用户可控

系统不再把 Agent 的判断完全隐藏。关键词、论文、笔记、分析、综述都可编辑；自动模式跑出的结果也可以回到分步模式继续修订。

### 14.2 渐进增强

复杂能力都配有 fallback：

```text
Embedding 失败 -> BM25
PDF 失败 -> 摘要
Skill 失败 -> 默认策略
笔记缺失 -> trace 兜底
分析缺失 -> 旧 Writer 流程
```

### 14.3 可观测

系统通过 trace、状态机、文件产物和评估脚本，把 Agent 行为从“模型自己做了什么”转化为可查看的工程过程。

### 14.4 可维护

后端从单文件拆成路由模块，工具集中注册，Session 文件结构固定，前后端 API 边界更清晰，降低了后续迭代成本。

## 15. 当前风险与后续建议

### 15.1 已接受风险

- 部分搜索工具默认禁用，因为外部 API 不稳定或限流严重；核心演示路径不依赖这些工具。
- 评估数据集仍可扩充；当前已有独立目录、脚本、数据库和 fallback。
- GitHub Actions 依赖 GitHub 镜像同步，若 GitLab dev 未合并到 master，GitHub 可能运行旧代码。

### 15.2 后续优先级

1. **Trace 摘要增强**：在轨迹页展示当前阶段、工具调用次数、错误类型统计、收录论文数。
2. **分步模式决策辅助**：论文卡片显示是否有笔记、是否被综述引用、是否适合对比/脉络/空白。
3. **评估数据补全**：增加自动模式、分析注入、Skill trace、Copilot 范围选择专项样例。
4. **前端回归测试**：为自动模式、分析卡片、Copilot 范围选择增加轻量端到端测试。
5. **部署可访问性**：若助教在线检查，需要保证 6.22-6.26 期间 Web 平台有公网或校内可访问链接。
