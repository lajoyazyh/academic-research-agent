## 1. 总览

迭代二的 Agent 更接近“一次性自动生成器”：用户输入主题后，系统自动规划、搜索、记录笔记并生成综述。这个流程虽然自动化程度高，但暴露出四个问题：

- 中间过程黑盒，用户无法在关键词、论文选择、笔记质量、综述结构等关键节点介入。
- 每次运行的数据相互孤立，缺少 Session 级持久化、断点继续和追加调研能力。
- 对话能力较弱，不能稳定支持多轮追问、基于资料的问答和基于反馈的修订。
- Pipeline 缺少可观测性，答辩时难以证明 Agent 如何调用工具、如何回退、如何处理失败。

迭代三的核心目标是把系统升级为“可交互、可追踪、可修订、可评估”的学术调研助手。最终形成了两种使用方式：

- **分步模式**：用户在规划、搜索、笔记、分析、综述等节点逐步确认和修改。
- **自动模式**：点击“自动进行”后，一键执行 `规划 -> 搜索 -> 笔记 -> 分析 -> 综述`，同时保留执行轨迹和中间产物。

## 2. 最终 Pipeline

### 2.1 运行阶段

```text
创建 Session
  -> 规划：生成关键词与检索计划
  -> 搜索：ReAct Agent 调用学术工具检索论文
  -> 笔记：RAGNoteGenerator 基于论文全文/摘要生成结构化笔记
  -> 分析：生成文献对比、研究脉络、研究空白
  -> 综述：将笔记和分析洞察注入 Writer，生成综述草稿
  -> 修订：用户通过编辑器或对话反馈继续修改
```

### 2.2 两种控制模式

分步模式保留用户参与关键判断的能力。用户可以编辑关键词、手动添加论文、选择哪些论文生成笔记、修改笔记、编辑分析卡片、再生成综述。

自动模式解决“我想直接跑到底”的需求。系统会自动执行完整 Pipeline，并在状态栏和轨迹区持续显示当前阶段。自动模式不是替代分步模式，而是同一套 Pipeline 的不同控制入口。

### 2.3 分析阶段并入主流程

迭代三后期将分析阶段正式并入主 Pipeline：

- 自动模式中，分析阶段位于笔记之后、综述之前。
- 分析结果保存到 `sessions/{session_id}/analysis/analysis_results.json`。
- 前端展示为三张可编辑 Markdown 卡片：文献对比、研究脉络、研究空白。
- 综述生成时会读取分析结果，将其作为额外写作上下文。
- 如果没有分析结果，Writer 保持旧流程，不影响兼容性。

这使分析结果不再只是页面展示，而是实际参与综述写作。

## 3. 架构优化

### 3.1 Session 状态管理

迭代三引入 `SessionManager`，每个研究任务都有独立目录：

```text
sessions/{session_id}/
├── metadata.json
├── plan/
├── papers/
├── notes/
├── analysis/
├── draft/
├── traces/
└── chats/
```

状态机覆盖：

```text
planning -> plan_confirmed -> searching -> search_complete
-> reviewing_notes -> writing -> reviewing_draft -> complete
```

主要收益：

- 中间产物持久化，刷新页面后不会丢失。
- 用户可以从不同阶段继续执行。
- 论文、笔记、分析、综述和对话历史被统一绑定到同一个 Session。
- 对卡住状态提供自动修复，降低演示时因异常状态导致的阻塞。

### 3.2 Web 后端模块化

早期 `web_app.py` 承担了大量路由和业务逻辑，后期拆分为：

```text
backend/routes/
├── deps.py
├── models.py
├── pages.py
├── session.py
├── chat.py
├── conversation.py
├── draft.py
├── agent.py
└── admin.py
```

拆分后，`web_app.py` 只负责应用初始化、依赖注入和路由注册。路由模块按功能聚合，降低了维护成本，也方便后续测试。

### 3.3 前端交互升级

前端从简单结果展示升级为 NotebookLM 风格控制台：

- 首页支持项目概览、最近活动、收藏和全局 Copilot。
- 控制台支持关键词、论文、笔记、分析、综述、轨迹等标签页。
- 笔记、分析卡片、综述均使用可编辑 Markdown 面板。
- 搜索和自动流程提供实时状态轮询。
- 轨迹区展示 Agent 的 thought/action/observation，支持解释运行过程。

## 4. Agent 与 Pipeline 能力增强

### 4.1 Plan + ReAct + Reflexion

系统保留并强化三类推理机制：

- **Plan-and-Execute**：先生成检索计划和关键词。
- **ReAct**：搜索阶段按 thought/action/action_input/observation 循环调用工具。
- **Reflexion**：对 JSON 错误、工具异常、连续失败、FINISH 前质量门禁进行反馈。

质量门禁要求 Agent 至少收录一定数量论文，避免过早 FINISH。

### 4.2 论文搜索与收录

主要改进：

- arXiv、OpenAlex、Crossref 等工具统一接入搜索阶段。
- Paper Register 将“论文登记 + PDF 下载”合并为单次动作，减少“下载了但没登记”或“登记了但没 PDF”的不一致。
- 搜索完成后扫描 traces 中的 arXiv ID，自动兜底下载 PDF。
- 支持用户手动添加论文、上传 PDF、删除论文、更新论文状态。

部分不稳定搜索工具默认禁用，这是主动规避限流和演示不确定性的策略，不视为功能缺失。

### 4.3 深度笔记生成

`RAGNoteGenerator` 用 RAG 方式生成结构化笔记：

- 将论文全文切块。
- 使用 Embedding 或 BM25 检索与各笔记章节最相关的片段。
- 生成研究背景、核心方法、实验设置、关键结果、消融分析、亮点与不足等结构化内容。
- PDF 提取失败时回退到摘要生成。
- Embedding 失败时回退到 BM25。

相比迭代二的自由笔记，迭代三笔记结构更稳定，也更适合后续综述写作。

### 4.4 Writer 与分析上下文

Writer 现在不仅使用笔记，还会读取分析阶段产物：

- `compare` 支撑方法对比。
- `lineage` 支撑研究脉络和技术演进。
- `gaps` 支撑研究空白和未来方向。

写作阶段仍保持兼容：没有分析结果时，只基于笔记生成综述。

### 4.5 Skills 自定义行为

Skills 允许用户为三个阶段配置自定义策略：

| 阶段 | 类型 | 作用 |
|------|------|------|
| 搜索 | `search` | 控制检索策略、数据源偏好、筛选标准 |
| 笔记 | `notes` | 控制笔记结构、关注维度、输出格式 |
| 综述 | `write` | 控制综述结构、写作风格、引用方式 |

采用双通道机制：

- Skill 存在且内容有效时，使用 Skill 策略。
- Skill 未配置、为空、被删除或加载失败时，自动回退默认策略。

2026-06-18 增强了 Skill 可观测性。search、notes、write 三个阶段都会记录 `SKILL_STATUS` trace，包含阶段、skill_id、skill_title、是否成功加载、是否回退默认及原因。这样答辩时可以直接在轨迹中证明 Skill 是否进入执行流程。

## 5. 对话与知识共享

### 5.1 Session 内对话

每个 Session 支持多个聊天会话，存储在：

```text
sessions/{session_id}/chats/
├── _index.json
└── conv_*.json
```

能力包括：

- 多轮历史保存。
- 普通问答和 Agent 修订共用统一入口。
- RAG 检索当前 Session 的论文内容。
- 对用户意图进行判定：普通问答、修订笔记、修订综述或需要澄清。
- 上下文窗口统计和压缩，避免长对话超出模型上下文。

### 5.2 全局 Copilot

首页全局 Copilot 支持跨 Session 知识问答：

- 扫描所有真实 Session 的论文摘要、笔记和综述草稿。
- 建立全局 TF-IDF/BM25 风格索引。
- 回答问题时引用相关 Session 的内容。
- 支持多轮 Copilot 对话历史。

2026-06-18 新增显式知识范围选择：

- `/api/knowledge/chat` 支持可选 `session_ids`。
- 不传 `session_ids` 时保持全局检索。
- 传入 `session_ids` 时只从选中 Session 的 chunks 中检索。
- 前端 Copilot 侧栏新增“知识范围”多选，默认选择全部真实 Session。

这使“跨对话知识共享”从隐式全局能力升级为用户可控能力。

## 6. 工具管理与可观测性

### 6.1 工具注册中心

工具通过 `core/tool_registry.py` 统一管理，并持久化到 `config/tools.json`。

能力包括：

- 工具按类别和 Pipeline 阶段标注。
- 支持启用/禁用。
- 支持参数配置。
- 前端工具管理面板可视化操作。

### 6.2 Trace 体系

Trace 记录 Agent 的运行过程：

- 搜索阶段记录 ReAct 的 thought/action/observation。
- 自动流程周期性同步 trace，前端可实时查看。
- Skill 状态作为 `SKILL_STATUS` 进入 trace。
- 常见错误通过 `error_type` 标记，为后续错误统计和答辩解释提供基础。

Trace 的答辩价值在于，它把“AI 随机执行”转化为“可解释、可追踪、可复盘”的过程。

## 7. 评估工作

### 7.1 evaluation 目录

同学已将评估部分整理到 `evaluation/`，主要内容包括：

```text
evaluation/
├── evaluation_tasks.csv
├── eval_platform.db
├── run_eval.py
├── requirements-eval.txt
├── requirements.txt
├── README.md
└── code/
```

其中：

- `evaluation_tasks.csv` 存放评估任务数据。
- `run_eval.py` 提供独立评估入口。
- `code/` 中包含服务层、数据访问层和评估方法。
- 已补充 fallback，使缺少部分外部评估依赖时仍可运行基本流程。

### 7.2 评估指标

文档和评估脚本主要关注：

| 指标 | 含义 |
|------|------|
| context_precision | 检索上下文的相关精度 |
| context_recall | 检索上下文覆盖相关信息的程度 |
| faithfulness_score | 生成内容对资料的忠实程度 |
| helpfulness_score | 回答或综述对用户的帮助程度 |
| relevance_score | 内容与问题/主题的相关性 |
| reasoning_score | 推理质量与结构化程度 |
| overall_score | 综合得分 |

### 7.3 阶段性评估结论

早期几轮评估显示：

- Session 和多轮对话显著提升了上下文精度。
- RAG 升级对召回和推理帮助最大。
- Skill 定制提升了可控性和忠实度，但也提醒我们要平衡“个性化策略”和“通用覆盖面”。
- 总体分数保持稳定，说明系统在功能复杂度提高的同时没有明显牺牲基础质量。

评测数据集后续如果继续补全，应重点补：

- 多 Session 知识共享问题。
- Skill 启用/回退可观测性问题。
- 分析阶段对最终综述质量的影响。
- 自动模式端到端成功率。

## 10. 当前风险与后续建议

### 已接受风险

- 部分搜索工具默认禁用，主要原因是外部 API 不稳定或限流严重，不影响核心流程达标。
- 评估数据集仍可继续扩充，当前已具备目录、脚本和基础数据。

### 后续优先级

1. **运行过程解释增强**：在 trace 面板聚合当前阶段、工具调用次数、错误类型、论文收录数。
2. **分步模式决策辅助**：在论文卡片中显示是否有笔记、是否被综述引用、是否适合对比/脉络/空白分析。
3. **评估数据补全**：增加自动模式、Skill trace、全局 Copilot 范围选择、分析注入综述的专项样例。
4. **前端回归测试**：针对自动模式、分析卡片、Copilot 范围选择增加轻量端到端测试。
