# 项目架构分析

## 一、项目概述

项目是一个 **Session 驱动的交互式学术文献调研平台**。用户输入研究主题，Agent 自主规划检索策略，从 arXiv / Semantic Scholar / Crossref / OpenAlex 等学术数据库搜索论文，阅读摘要和全文，生成结构化笔记，最终产出万字级文献综述。

核心流程分为四个阶段：

```
Plan（规划） → Search（搜索） → Notes（笔记生成） → Write（撰写综述）
```

两条执行路径并存：
1. **CLI 路径**（`main.py`）：`run_agent_pipeline()` 一次性执行全流程，适合脚本化
2. **Web 路径**（`web_app.py`）：通过 FastAPI 端点分阶段执行，支持断点/继续，适合交互式平台

---

## 二、核心模块

### 2.1 Agent 主循环 — `core/agent.py`（BaseAgent）

整个系统的**大脑**，融合了三种推理框架：

- **Plan-and-Execute**：`_generate_plan()` 让 LLM 先产出研究计划（关键词、数据库选择、备选策略、预期深度）
- **ReAct 循环**：`run()` 主循环中，思考 → 调用工具 → 观察结果 → 再思考
- **Reflexion 反思**：五层自我纠错——
  - L1: JSON 解析失败 / 未知工具 / 运行时异常即时反馈
  - L2: 连续 3 次同类错误触发深度反思提示
  - L3: FINISH 前自主质检（Self-Critique）
  - L4: PDF 自动兜底下载
  - L5: 论文列表自动传递到前端
- **质量门禁**：FINISH 前检查 `paper_register` 调用次数 ≥ `AGENT_MIN_PAPERS`（默认 3）
- **Token 防爆**：observation 超过 4500 字符自动截断

### 2.2 CLI 入口 — `main.py`

- `run_agent_pipeline()`：传统一次性全流程（规划→搜索→笔记→综述），创建独立 `documents/` 目录
- `run_plan_only()`：阶段 1，生成初始规划 + 提取关键词候选项
- `run_search_only()`：阶段 2，使用确认后的关键词执行搜索，集成了 `PaperRegisterTool`（下载 PDF + 登记论文）
- `run_write_from_notes()`：阶段 3，基于笔记撰写/重写综述，支持用户反馈迭代
- `run_agent_pipeline_session()`：Session-aware 统一入口，支持从任意断点继续
- `_build_initial_plan()`：生成检索计划（关键词拆分、数据源优先级、回退策略）
- `_extract_keywords_from_plan()`：用 LLM 从规划文本中提取结构化关键词三元组

### 2.3 Web 服务 — `web_app.py`

FastAPI 应用，提供完整的 REST API 和前端服务。关键端点按功能域划分：

**Agent 执行**：
- `/run/plan`：调用 `run_agent_pipeline_session(start_phase="plan")`，保存规划+关键词到 Session
- `/run/search`：后台线程执行，通过 `RUNS` 全局字典 + 轮询机制实现实时状态推送
- `/run/status`：前端轮询搜索进度
- `/run/cancel`：通过 `_stop_flag` 中断后台搜索
- `/run/notes`：RAG 深度笔记生成（调用 `RAGNoteGenerator`）
- `/run/write`：撰写综述
- `/run/auto`：一键自动模式（规划→搜索→笔记→综述全流程）

**对话与聊天**（详见[第四章](#四对话子系统详解)）：
- `/chat`：统一聊天入口，支持普通问答 + AI 判意修订
- `/chat/stream`：SSE 流式聊天端点，逐 token 推送 + 引用标注

**Session 与论文管理**：创建/列表/删除 Session、论文 CRUD、笔记与草稿读写、状态转移等。

## 三、数据流与状态管理

### 3.1 Session 数据目录

```
sessions/{session_id}/
├── metadata.json       # 会话元数据 + 状态
├── plan/               # 初始规划
├── papers/             # PDF + papers_list.json
├── notes/              # 笔记草稿 + 编辑历史
├── draft/              # 综述草稿多版本
└── traces/             # 执行轨迹（追加模式）
```

### 3.2 状态机

```
planning → plan_confirmed → searching → search_complete
→ reviewing_notes → writing → reviewing_draft → complete
```

- 任意状态可跳回 `searching`（追加调研）
- 加载时自动检测卡住状态并修复

## 四、对话子系统详解

对话模块是 Web 平台的核心交互入口，经过五个阶段优化后，已从简单问答升级为功能完备的智能对话系统，支持多轮记忆、混合检索、流式输出和引用标注。

### 4.1 总体架构

```
用户消息 → /api/sessions/{id}/chat 或 /chat/stream
  │
  ├── chat_mode="normal" → 直接走「普通问答」分支（不做意图判意）
  │
  └── chat_mode="agent"
        ├── 前端确认修订 (confirmed_revision) → 直接执行修订
        ├── 显式 /修订 指令 → _parse_explicit_chat_revision() → 直接执行修订
        ├── 轻量级关键词预判 → _quick_intent_check() → 规则命中则直接判定
        ├── AI 意图判意 → _infer_chat_revision_ai() → LLM 分类（仅规则无法判定时）
        └── 普通问答 → _build_chat_answer() 或流式端点
```

### 4.2 两种模式对比

| 维度 | 对话模式 (normal) | Agent 模式 (agent) |
|------|-------------------|-------------------|
| 意图判意 | 无，全部视为提问 | 三级判意：确认 > 显式命令 > AI 推断 |
| 修订能力 | 不支持 | 支持笔记修订 + 综述重写 |
| 确认机制 | 无 | AI 推断的修订需用户二次确认 |
| RAG 检索 | ✅ HybridRetriever | ✅ HybridRetriever |
| 对话历史 | ✅ 最近 12 轮 | ✅ 最近 12 轮 |
| 上下文截断 | ✅ 智能配额管理 | ✅ 智能配额管理 |
| 流式输出 | ✅ SSE 逐 token | ✅ SSE 逐 token |
| 引用标注 | ✅ citations 数组 | ✅ citations 数组 |
| 适用场景 | 阅读时随意提问 | 需要修改笔记/综述时 |

### 4.3 普通问答 Pipeline（normal 模式）

```
用户消息
  │
  ▼
Step 1: 确定 conv_id（若未传或不存在 → 自动创建默认对话）
  │
  ▼
Step 2: _build_chat_answer()
  ├── 2a. 确定当前论文（current_paper_id 或第一篇 accepted）
  ├── 2b. RAG 检索 (HybridRetriever: BM25 + Embedding 双路召回 → Top-5)
  ├── 2c. 加载对话历史（最近 12 轮，格式: "用户：... / 助手：..."）
  ├── 2d. 上下文窗口管理 (_manage_context_window)
  │       配额: 笔记 40% / 草稿 30% / 摘要 10% / RAG 10% / 历史 10%
  ├── 2e. 构建 Prompt + LLM 调用
  │       system: 学术助理角色设定
  │       user: 主题/论文/摘要/笔记/草稿/RAG/历史/问题
  └── 2f. Fallback: _build_chat_reply()（LLM 失败时基于摘要/笔记的模板回复）
  │
  ▼
Step 3: 保存对话记录 (_save_chat_exchange → append_conversation_messages)
  │
  ▼
返回 JSON: { reply, note, action: "chat", rag_status, rag_count }
```

### 4.4 Agent 模式 Pipeline（含意图判意）

Agent 模式在问答基础上增加了**三级意图判意**，决定用户消息是"提问"还是"修改请求"。

```
用户消息
  │
  ▼
Step 1: 确定 conv_id（同上）
  │
  ▼
Step 2: 三级意图判意 (revision detection)
  │
  ├── 2a. 前端确认修订 (confirmed_revision)
  │     用户在上轮 AI 询问"是否确认修改"后点了确认 → 直接执行修订
  │
  ├── 2b. 显式命令解析 (_parse_explicit_chat_revision)
  │     "/修订 笔记 删除第二部分" → target=report
  │     "/修订 综述 重写引言"     → target=review
  │     → 直接执行修订 (无需确认)
  │
  └── 2c. AI 意图判意 (_infer_chat_revision_ai)
        │
        ├── 2c-i. 轻量级关键词预判 (_quick_intent_check)
        │     修改词 ≥2 且 疑问词 =0 → "revise"（直接构造 revision）
        │     疑问词 ≥1 且 修改词 =0 → "chat"（走问答）
        │     其他情况 → None（不确定，走 LLM）
        │
        └── 2c-ii. LLM 意图分类（仅规则无法判定时）
              intent: revise | chat | clarify
              confidence ≥ 0.6 → 执行
              confidence < 0.6 → 降级为问答
              LLM 失败 → 降级为问答
  │
  ▼
Step 3: 根据 revision 结果分流
  ├── revision == None → 走「普通问答」分支
  ├── revision.intent == "clarify" → 返回澄清提示
  ├── revision.source == "ai" → 返回确认请求（前端弹出对话框）
  └── revision.source == "explicit" | "confirmed" → 直接执行修订
        target == "report" → revise_notes_phase()
        target == "review" → save_feedback() + run_write()
  │
  ▼
Step 4: 保存对话记录 + 返回结果
```

### 4.5 流式端点（SSE）

```
用户消息
  │
  ▼
1. 确定 conv_id + 当前论文
2. RAG 检索 (HybridRetriever)
3. 加载对话历史（最近 12 轮）
4. 上下文窗口管理 (_manage_context_window)
5. 构建 system + user prompt
  │
  ▼
SSE 流式输出:
  data: {"token": "这"}
  data: {"token": "篇"}
  data: {"token": "论文"}
  ... (逐 token 推送，打字机效果)
  data: {"citations": [{ id: "[1]", paper_id, title, page, snippet }, ...]}
  data: {"done": true, rag_status: "used", rag_count: 3}
  │
  ▼
保存对话记录 (异步)
```

### 4.7 改善历程

对话模块经历了五个阶段的迭代优化：

| 薄弱点（改善前） | 改善措施 | 阶段 |
|------|------|------|
| 无多轮对话记忆 | `_build_chat_answer()` 注入最近 12 轮对话历史，支持指代消解 | 一 |
| 仅 BM25 检索 | 新增 `HybridRetriever`：BM25 + Embedding 双路召回加权融合 | 二 |
| 粗暴截断 2000 字符 | `_manage_context_window()` 智能配额分配：笔记 40%/草稿 30%/摘要 10%/RAG 10%/历史 10% | 三 |
| 意图判意额外 LLM 调用 | `_quick_intent_check()` 关键词规则预判，命中则跳过 LLM | 四 |
| 错误处理静默 | 规范化错误处理：LLM 失败有 fallback 模板，日志记录完善 | 四 |
| 无流式输出 | `LLMClient.chat_stream()` + SSE 端点，逐 token 推送打字机效果 | 五 |
| 无引用标注 | RAG 结果附带 `citations` 数组（编号、标题、页码、原文片段） | 五 |

改善前后整体对比：

| 维度 | 改善前 | 改善后 |
|------|--------|--------|
| 多轮对话 | ❌ 无状态，无法理解指代 | ✅ 最近 12 轮历史注入 prompt |
| 检索质量 | ⚠️ 仅 BM25 关键词匹配 | ✅ BM25 + Embedding 双路融合 |
| 上下文利用 | ❌ 粗暴截断 2000 字符 | ✅ 智能配额分配，按比例截断 |
| 响应延迟 | ⚠️ 每次额外 LLM 判意调用 | ✅ 规则优先，减少不必要调用 |
| 用户体验 | ❌ 等待完整回复，无进度感 | ✅ SSE 流式逐 token 输出 |
| 可追溯性 | ❌ 无引用来源 | ✅ citations 数组标注原文出处 |
| 错误鲁棒性 | ❌ 静默 pass，难以调试 | ✅ Fallback 模板 + 日志记录 |
| 修订能力 | ✅ 已支持 | ✅ 保持，三级判意更精准 |

## 五、关键设计决策

1. **`paper_register` 一体化工具**：解决了之前"下载 PDF 但未注册"的数据一致性问题
2. **后台线程 + 轮询**：搜索阶段用 `threading.Thread` + `RUNS` 全局字典实现非阻塞执行
3. **Trace 追加模式**：多次搜索自动追加，带 SECTION 分隔，前端 TOC 导航
4. **质量门禁**：FINISH 前强制检查论文数量，防止 Agent 过早结束
5. **兜底策略**：笔记生成失败时从 traces 自动提取，综述撰写有 fallback 大纲
6. **三级意图判意**：确认 > 显式命令 > AI 推断，兼顾安全性与智能化
7. **HybridRetriever 双路召回**：BM25 保证关键词匹配精度，Embedding 捕获语义相关性

## 七、总结

项目处于**迭代三完成、可正常运行**的状态。核心创新在于：

- **Session 驱动的交互式调研**：用户可在关键节点参与、编辑、反馈
- **三种推理框架融合**：Plan-and-Execute + ReAct + Reflexion
- **智能对话系统**：多轮记忆 + 混合检索 + 流式输出 + 引用标注 + 意图判意

当前最需要关注的是**性能优化**（PDF 提取、API 限流）和**代码重构**（`web_app.py` 过长、模块解耦）。
