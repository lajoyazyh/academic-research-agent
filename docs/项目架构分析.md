## 一、总体架构

### 1.1 系统定位

Academic Research Agent 是一款**Session 驱动的交互式学术文献调研平台**，也是一个**多 Agent 协作系统**。给定一个研究主题，系统自主完成关键词规划、多源学术搜索、论文筛选、深度笔记生成和万字级综述撰写。用户可在每个关键决策点介入编辑或反馈，与 Agent 协作完成高质量文献调研。

### 1.2 技术栈

| 层级 | 技术 | 用途 |
|------|------|------|
| **后端框架** | FastAPI + Uvicorn | Web API 服务，REST 端点 |
| **LLM** | ZhipuAI glm-4-flash | 推理、规划、生成 |
| **Embedding** | ZhipuAI embedding-2 | 向量语义检索（笔记生成） |
| **PDF 处理** | PyMuPDF (fitz) | PDF 文本提取 |
| **全文检索** | sklearn BM25 + numpy | 对话 RAG 段落召回 |
| **前端** | 原生 JavaScript ES6+ + Vditor | 交互式 Web UI |
| **测试** | pytest | 20 个单元测试用例 |
| **配置管理** | python-dotenv + JSON | 环境变量 + 工具配置持久化 |

### 1.3 多 Agent 谱系

系统包含**五种 Agent**，分布在两条运行路径中：

│  │  ReAct 循环   │  │  RAG 流水线   │  │大纲+逐节+RAG │           │
│  │ 10工具 4源   │  │ Emb+ 6节生成  │  │关键词段落检索│           │
│  └──────────────┘  └──────────────┘  └──────────────┘           │
│                                                                  │
│  ┌──────────────┐  ┌──────────────────────────────┐             │
│  │  Chat Agent  │  │      Copilot Agent           │             │

### 1.5 总体数据流

```
用户输入主题
    │
    ▼
[Plan]      LLM 生成关键词规划
    │
    ▼  (用户确认关键词)
[Search]    Search Agent: ReAct 循环 → 多源搜索 → 下载 PDF
    │
    ▼  (用户审核论文列表)
[Notes]     Notes Agent: PDF 全文 → Embedding → 逐节生成 6 段式笔记
    │
    ▼  (用户编辑笔记)
[Write]     Writer Agent: 大纲 → RAG 段落提取 → 逐节撰写 → 合并
    │
    ▼  (用户反馈修改)
[Analysis]  (可选) 文献对比 / 研究脉络 / 空白发现
    │
    ▼
最终交付物：万字级文献综述 + 深度分析报告
```

### 1.6 Web 应用解耦架构（v3.1）

为提升代码可维护性，`web_app.py` 从单文件 ~2600 行拆分为模块化路由结构：

```
web_app.py (90 行)              ← 精简入口：初始化 + 注册路由
backend/routes/
├── deps.py                     ← 共享依赖（路径常量、全局管理器注入）
├── models.py                   ← 共享 Pydantic 模型（20+ 数据类）
├── pages.py (247 行)           ← 页面路由、收藏夹、历史、PDF 服务
├── session.py (630 行)         ← Session/论文/笔记管理（prefix=/api/sessions）
├── chat.py (780 行)            ← 聊天系统：上下文窗口、意图判定、流式、压缩
├── conversation.py (65 行)     ← 多会话聊天管理
├── draft.py (46 行)            ← 综述草稿管理
├── agent.py (807 行)           ← Agent 执行：规划/搜索/笔记/综述/自动/分析
└── admin.py (339 行)           ← 管理 API：工具/知识库/Copilot/Skills
```

**关键设计**：
- 每个路由模块使用 `APIRouter(prefix=...)` 独立管理路径前缀
- `deps.py` 通过 `init_deps()` 延迟注入全局管理器，避免循环引用
- `models.py` 集中管理所有 Pydantic 模型，消除重复定义

## 二、Agent 核心引擎 (core/agent.py)

### 2.1 BaseAgent 设计理念

`BaseAgent` 是整个系统的核心推理引擎。它不绑定任何特定领域知识，只负责通用的思维-行动-观察循环。所有业务逻辑通过**注入工具列表**实现，遵循"工具即能力"的设计原则。

```python
class BaseAgent:
    def __init__(self, tools: List[BaseTool], max_loops: int = 5):
        self.llm = LLMClient()
        self.tools: Dict[str, BaseTool] = {tool.name: tool for tool in tools}
        self.max_loops = max_loops
        self.traces = []         # 执行轨迹
        self.error_history = []  # 连续错误统计
```

### 2.2 三种推理框架的有机融合

BaseAgent 的核心 `run()` 方法融合了三种经典 AI Agent 推理范式：

```
┌─────────────────────────────────────────────────┐
│            BaseAgent.run(user_query)             │
│                                                  │
│  ┌─────────────────────────────────────────┐    │
│  │  阶段 0: Plan-and-Execute               │    │
│  │  _generate_plan()                       │    │
│  │  LLM 产出一个显式研究计划                │    │
│  └─────────────────────────────────────────┘    │
│                      ▼                           │
│  ┌─────────────────────────────────────────┐    │
│  │  阶段 1-N: ReAct 循环                   │    │
│  │  for i in range(max_loops):             │    │
│  │    Thought → Action → Observation       │    │
│  │    1. LLM 推理 (强制 JSON 格式)         │    │
│  │    2. 工具调度并执行                    │    │
│  │    3. 观察记录 + trace                  │    │
│  │    4. Reflexion 反思检查               │    │
│  └─────────────────────────────────────────┘    │
│                      ▼                           │
│  ┌─────────────────────────────────────────┐    │
│  │  出厂前检查: 质量门禁                   │    │
│  │  paper_register 调用次数 ≥ MIN_PAPERS?   │    │
│  └─────────────────────────────────────────┘    │
└─────────────────────────────────────────────────┘
```

#### Reflexion 五层自纠错

| 层次 | 触发条件 | 机制 |
|------|---------|------|
| **L1: 即时反馈** | JSON 解析失败 / 工具不存在 / 运行时异常 | 错误信息注入下轮 LLM 上下文 |
| **L2: 深度反思** | 连续 3 次同一类型错误 | 注入反思提示：分析根因、提出替代方案 |
| **L3: 自主质检** | FINISH 前且笔记数 ≥ 3 | LLM 逐篇审查笔记完备性 |
| **L4: PDF 兜底下载** | Agent 主循环结束后 | 扫描轨迹批量下载 PDF |
| **L5: 论文列表传递** | Agent 主循环结束后 | 自动收集已下载 PDF 列表传给前端 |

#### 质量门禁

FINISH 前强制检查 `paper_register` 调用次数 ≥ `AGENT_MIN_PAPERS`（默认 3），不满足则拦截。

### 2.3 工具基类

```python
class BaseTool:
    name: str           # 工具名
    description: str    # 功能描述
    parameters: dict    # 参数 schema

    def execute(self, **kwargs) -> str:
        """执行工具调用，返回 observation 字符串"""
```

工具返回值必须是字符串（observation），该字符串在下一轮 ReAct 中作为上下文注入 LLM。

## 三、Search Agent — ReAct 多源检索

### 3.1 Agent 定位

Search Agent 是系统中**唯一使用完整 ReAct 循环的 Agent**。它持有 10 个学术搜索工具，在 Plan-and-Execute 规划的指导下自主探索 4 个学术数据库，收集论文元数据并下载 PDF。

### 3.2 推理循环

```
用户确认的关键词 + 初始规划
    │
    ▼
run_search_only()  [main.py]
    │
    ├── 1. 创建工具实例 (10个搜索工具 + paper_register)
    │     · arxiv_search / arxiv_fetch
    │     · semantic_scholar_search / semantic_scholar_fetch
    │     · crossref_search / crossref_fetch_doi
    │     · openalex_search
    │     · paper_register (下载+登记)
    │
    ├── 2. 构建 research_query
    │     包含: 研究目标、完成标准、禁止行为列表、
    │           自主规划要求、数据库优先级、关键经验
    │
    ├── 3. BaseAgent(tools=search_tools).run(research_query)
    │     ┌─ ReAct 第1轮: Thought→ arxiv_search("keyword") → Obs: 5篇论文
    │     ├─ ReAct 第2轮: Thought→ arxiv_fetch(paper_id)  → Obs: 完整元数据
    │     ├─ ReAct 第3轮: Thought→ paper_register(...)   → Obs: 已收录
    │     ├─ ReAct 第4轮: Thought→ openalex_search(...)  → Obs: 3篇论文
    │     └─ ReAct 第N轮: Thought→ FINISH                 → 退出
    │
    └── 4. PDF 自动下载 (扫描轨迹中的 arXiv ID)
```

### 3.3 查询构建策略

**硬性完成标准**：
1. 搜索到至少 3 篇论文，获得标题+作者+摘要
2. 至少使用两个不同的学术数据库
3. 收集够 3 篇高质量论文后必须立即 FINISH

**严禁行为列表**（精准约束，避免常见失败模式）：
- 禁止同一关键词反复搜索同一数据库超过 2 次
- 禁止连续 3 轮用相同参数调用同一工具
- 禁止搜索到好论文后无意义重复

**数据库选择优先级**：
- CS/AI/理工科 → 优先 arXiv（限流较松，结果完整）
- 医学/社科/跨学科 → 优先 OpenAlex（覆盖广）
- Semantic Scholar → 仅作后备补充（限流严格，60% 返回 429）

### 3.4 工具集

| 工具 | API | 功能 |
|------|-----|------|
| `arxiv_search` | arXiv API | 关键词搜索，返回标题+作者+摘要 |
| `arxiv_fetch` | arXiv API | 按 ID 获取完整元数据 |
| `arxiv_download_pdf` | arXiv PDF | 下载 PDF 到本地 |
| `arxiv_pdf_reader` | PyMuPDF | 下载+解析 PDF 全文 |
| `semantic_scholar_search` | Semantic Scholar API | 关键词搜索 + 引用数 |
| `semantic_scholar_fetch` | Semantic Scholar API | 按 ID 获取详情 + 引用关系 |
| `crossref_search` | Crossref API | 关键词/标题检索 |
| `crossref_fetch_doi` | Crossref API | 按 DOI 补全期刊/年份 |
| `openalex_search` | OpenAlex API | 跨学科搜索 |
| `paper_register` | 内部工具 | 审核摘要→下载 PDF + 登记论文 |

### 3.5 工程优化

- **全局限流器**：arXiv 最小 5s 间隔，429 指数退避重试
- **查询变体降级**：零结果时 Agent 自主替换同义词/上位词
- **容错兜底**：`append_note` 未执行时从 traces 自动提取笔记

## 四、Notes Agent — RAG 深度笔记

### 4.1 Agent 定位

Notes Agent 采用**固定 RAG 流水线**而非 Agent 循环。设计理由：笔记生成是高度结构化的任务（提取→分块→检索→生成×6节），不需要 Agent 级别的自主决策。固定流水线更高效、更稳定，且不消耗 LLM 推理 token。

### 4.2 RAG 流水线

```
选中的论文 (paper_ids)
    │
    ▼
RAGNoteGenerator.generate(pdf_path, title, abstract, topic)
    │
    ├── 1. PDF 文本提取 (PyMuPDF)
    │     读取所有页面 → 按结构分块 (chunk_size ≈ 500 字)
    │
    ├── 2. 向量化 (ZhipuAI Embedding API)
    │     每个 chunk → 1024 维向量 → 构建向量索引
    │
    ├── 3. 逐节检索+生成 (6 维度循环)
    │     for each dimension [研究背景, 核心方法, 实验设置,
    │                          关键结果, 消融分析, 亮点不足]:
    │       ① 构建维度查询向量 (关键词嵌入)
    │       ② 余弦相似度计算 Top-5 相关段落
    │       ③ 原文段落 + 维度写作要求 → LLM 生成笔记
    │       ④ 检查字数 ≥ min_words，不足则扩充
    │
    └── 4. 拼接 6 段 → 完整笔记
```

### 4.3 三级 Fallback

| 级别 | 通道 | 触发条件 |
|------|------|---------|
| **L1** | Embedding 向量检索 | 默认，质量最高 |
| **L2** | BM25 关键词检索 | Embedding API 不可用 |
| **L3** | 标题+摘要+前 5 页 | PDF 不可用 |

## 五、Writer Agent — 大纲+逐节+RAG 撰写

### 5.1 Agent 定位

Writer Agent 采用**大纲规划 + 逐节撰写**的分步流水线，并引入了与 Notes Agent 相似的 **RAG 段落提取**能力。设计理由：综述撰写有清晰的结构（引言→方法对比→实验分析→局限），不需要 Agent 循环。

### 5.2 流水线架构

```
聚合后的笔记内容 (notes_content)
    │
    ▼
run_write_from_notes()  [main.py]
    │
    ├── 1. _build_writer_outline()
    │     【大纲规划】输入: 主题 + 笔记 → 输出: 4 节 Markdown 大纲
    │     LLM 调用 1 次
    │
    ├── 2. _compose_review_by_sections()
    │     【逐节撰写 + RAG 段落提取】
    │     for section in [引言与背景, 核心方法对比, 实验结果分析, 局限与未来]:
    │       ├── _extract_relevant_paragraphs(notes, section_title)
    │       │     ① 用章节关键词映射从笔记中检索匹配段落
    │       │     ② 评分: 命中数 + 标题精确匹配 + 密度
    │       │     ③ 返回 Top-6 高分段落
    │       ├── section_prompt = 主题 + 大纲 + 已完成章节 + 相关段落
    │       └── LLM 生成该节
    │     LLM 调用 4 次
    │
    └── 3. 合并输出
          大纲(引用格式) + 间隔线 + 逐节正文
```

### 5.3 Writer RAG 段落提取

与 Notes Agent 的 Embedding 检索不同，Writer Agent 使用**确定性关键词匹配**：

- 每个章节预定义关键词映射（如"核心论文方法对比"→ method/approach/model/architecture/...）
- 将笔记按 `##` 标题拆分为段落
- 对每段计算命中分数 + 标题精确匹配 + 密度加权
- 返回 Top-6 高分段落注入 prompt

**效果**：每节 prompt 从 ~15K chars 降到 ~3K chars，LLM 聚焦于最相关的笔记段落，减少无关内容干扰，节省 token 消耗。

### 5.4 章节关键词映射

| 章节 | 检索关键词 |
|------|-----------|
| 引言与背景 | background, problem, challenge, limitation, gap, motivation |
| 核心论文方法对比 | method, approach, proposed, architecture, model, framework |
| 实验结果分析 | experiment, result, performance, evaluation, benchmark, accuracy |
| 局限性与未来方向 | limitation, future, drawback, challenge, 改进, 展望 |

### 5.5 反馈重写

支持用户反馈驱动的多轮重写，用户反馈和上一版草稿共同注入 prompt，LLM 同时参考笔记和用户意见生成新版本。所有历史版本保留在 `sessions/{id}/draft/draft_v{N}.md`。

## 六、Chat Agent — 控制台对话与修订协同

### 6.1 Agent 定位

Chat Agent 是 Web 平台**控制台内的交互 Agent**，底部对话区同时承担两种功能：
- **普通问答**：用户在浏览论文/笔记/综述时提问
- **修订协同**：用户在 Agent 模式下请求修改笔记或综述

### 6.2 推理循环

Chat Agent 遵循**感知→检索→推理→回答**的单步隐式 ReAct 模式（每一步都是完整的检索+推理，但不驱动工具调用）：

```
用户消息
    │
    ▼
POST /api/sessions/{id}/chat
    │
    ├── chat_mode="normal" → 直接问答
    │
    └── chat_mode="agent"
          ├── 前端确认修订 → 执行修订
          ├── 显式 /修订 指令 → 执行修订
          ├── 关键词规则预判 → 修订或问答
          └── AI 意图分类 → LLM 判定 revise/chat/clarify
    │
    ▼
_build_chat_answer()
    ├── HybridRetriever (BM25 + Embedding 双路召回) → Top-5 段落
    ├── 对话历史 (最近 12 轮)
    ├── _manage_context_window() 配额分配
    ├── LLM 推理 → 生成回答
    └── 保存对话记录
```

### 6.3 两种模式对比

| 维度 | 对话模式 (normal) | Agent 模式 (agent) |
|------|-------------------|-------------------|
| 意图判意 | ❌ 无 | ✅ 三级判意（确认 > 显式 /修订 > AI 推断） |
| 修订能力 | ❌ | ✅ 笔记修订 + 综述重写 |
| 确认机制 | — | AI 推断需用户二次确认 |
| RAG 检索 | ✅ HybridRetriever | ✅ HybridRetriever |
| 对话历史 | ✅ 最近 12 轮 | ✅ 最近 12 轮 |
| 流式输出 | ✅ SSE | ✅ SSE |

### 6.4 上下文窗口管理

| 上下文类型 | 配额 |
|-----------|------|
| 用户消息 | 不限 |
| RAG 检索结果 | ≤ 30,000 字符 |
| 对话历史 (12 轮) | ≤ 15,000 字符 |
| 笔记 | ≤ 20,000 字符 |
| 草稿 | ≤ 20,000 字符 |

## 七、Copilot Agent — 全局知识整合

### 7.1 Agent 定位

Copilot Agent 是**首页侧边栏的全局 Agent**，能够跨所有 Session 进行知识整合。与 Chat Agent（绑定单个 Session）不同，Copilot 拥有全局视野。

### 7.2 推理循环

```
用户消息
    │
    ▼
POST /api/knowledge/chat
    │
    ├── 1. 全局知识库检索
    │     GlobalKnowledgeBase.search(query, top_k=8)
    │     检索所有 Session 的论文摘要、笔记、草稿
    │
    ├── 2. 获取全局统计信息
    │     总 Session 数 / 总论文数 / 总笔记字数 / 总草稿数
    │
    ├── 3. 构建 RAG 上下文
    │     每个检索结果标注来源 (Session 主题 + 内容类型)
    │
    ├── 4. LLM 推理 → 生成回答
    │     system: "学术研究 Copilot 助手"
    │     user: 全局统计 + 检索资料 + 用户问题
    │
    └── 5. 保存到 Copilot 会话历史
          sessions/.copilot_sessions/{id}.json
```

### 7.3 与 Chat Agent 的区别

| 维度 | Chat Agent (控制台) | Copilot Agent (首页) |
|------|---------------------|---------------------|
| **作用域** | 单个 Session | 所有 Session |
| **检索源** | 当前 Session 的 PDF 段落 | 所有 Session 的摘要/笔记/草稿 |
| **修订能力** | ✅ 笔记 + 综述 | ❌ 不支持 |
| **多轮对话** | ✅ Conversation CRUD | ✅ Copilot 会话独立管理 |
| **触发位置** | 控制台底部对话栏 | 首页侧边栏 |

## 八、Analysis Agent — 深度分析

### 8.1 Agent 定位

Analysis Agent 是写入阶段之后的**可选增强**。基于已有笔记和论文元数据，产出三种分析报告：

| 工具 | 函数 | 产出 |
|------|------|------|
| **文献对比** | `compare_papers(topic, notes, papers)` | 方法对比表格 + 优缺点矩阵 |
| **研究脉络** | `trace_lineage(topic, notes, papers)` | 时间线 + 技术演进路径 |
| **空白发现** | `find_gaps(topic, notes, papers)` | 已覆盖子领域 + 未来方向建议 |

### 8.2 实现方式

三项分析不涉及对外 API，完全是 LLM 驱动的文本分析：
1. 从 `papers_list.json` 提取论文元数据
2. 构造结构化 prompt（含论文摘要 + 笔记 + 分析要求）
3. LLM 生成 Markdown 报告
4. 保存到 `sessions/{id}/analysis/analysis_results.json`

## 十三、总结

### 13.1 五种 Agent 架构对比

| Agent | 推理模式 | 是否 ReAct | 检索方式 | 适用场景 |
