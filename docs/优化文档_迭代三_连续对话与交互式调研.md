## 现状分析

### 迭代二的问题

当前系统采用**单次全流程执行**：

```
用户输入主题
    ↓
Plan（规划关键词）[无用户交互]
    ↓
Researcher（搜索→读取→笔记）[黑盒执行]
    ↓
Writer（综述撰写）[黑盒执行]
    ↓
用户获取最终输出
```

**核心痛点**：
- 中间过程完全黑盒，用户无法干预
- 一旦前期搜索策略有偏差，后续全功亏一篑
- 用户有新发现的论文无法插入
- 笔记生成后无法修改就进入撰写
- 综述出来后若有意见，无法基于现有笔记快速迭代

---

## 优化目标

| 目标 | 具体表现 |
|------|---------|
| **1. 连续对话支持** | 同一个会话内，用户可多次交互而无需重新启动整个流程 |
| **2. 关键步骤可见化** | 在关键决策点（关键词、论文列表、笔记草稿、初稿）暂停，等待用户反馈 |
| **3. 混合调研来源** | Agent 搜索结果 + 用户上传论文，两种来源混合编辑 |
| **4. 草稿可修改** | 笔记和初稿支持用户直接编辑或让 Agent 基于意见重写 |
| **5. 分阶段重执行** | 用户修改后，只需重新执行后续阶段，无需全部重来 |

## 可行性评估

### ✅ 高优先级（强烈建议实现）

#### 1.1 **关键词拆解确认**
- **合理性**: ⭐⭐⭐⭐⭐
  - 用户最清楚自己的研究意图和可能的偏差
  - Early validation 能避免后续大量无关论文的检索
  - 成本低，收益大
- **技术难度**: 低
- **实现方式**:
  - Plan 阶段后，展示生成的关键词三元组（中文 → 英文 → 同义词）
  - 用户可编辑、删除、新增关键词，确认后才进入 Search

#### 1.2 **草稿笔记审核 + 修改**
- **合理性**: ⭐⭐⭐⭐⭐
  - Researcher 生成的笔记质量直接影响最终综述质量
  - 用户可能发现 Agent 遗漏的关键信息或错误理解
  - 支持改进后的笔记能显著提升最终交付物
- **技术难度**: 中等
- **实现方式**:
  - Researcher 完成后，展示所有笔记（以 Markdown 列表形式）
  - 用户可在 Web UI 中在线编辑、删除某篇、或上传新论文补充笔记
  - 编辑后确认，才进入 Writer 阶段

#### 1.3 **用户上传补充论文**
- **合理性**: ⭐⭐⭐⭐⭐
  - 现实调研中，用户往往有一些"必看论文"清单
  - 支持用户手动补充能覆盖"混合检索"的常见需求
- **技术难度**: 中等
- **实现方式**:
  - 支持三种输入：标题（自动搜索）、DOI/arXiv ID（自动fetch）、PDF 直传（自动提取+生成笔记）
  - 上传后的论文与 Agent 搜索结果混合显示，用户可统一编辑

#### 1.4 **最终综述可修改+重新撰写**
- **合理性**: ⭐⭐⭐⭐
  - 综述呈现后，用户可能有stylistic 或结构化的意见
  - 支持基于意见快速 re-write 比从零开始快得多
- **技术难度**: 低（复用 Writer 的逻辑，增加反馈上下文）
- **实现方式**:
  - 综述呈现后，用户可标记"需修改段落"或写整体反馈
  - 用反馈作为 System Prompt 的补充上下文，让 LLM 重新生成

### ✅ 高优先级（进一步完善）

#### 2.1 **论文智能确认 + 自动关闭**
- **合理性**: ⭐⭐⭐⭐⭐
  - 防止自动纳入无关论文，保证质量
  - 提供"可选审查"，避免强制交互疲劳
  - 自动关闭防止长时间卡住
- **技术难度**: 低
- **实现方案**:
  - 论文列表页面新增"启用审查模式"toggle
  - 启用后，每篇论文带上"✓ 接受 / ✗ 拒绝 / ? 稍后决定"按钮
  - **自动关闭策略**：如果用户 1 分钟内无交互，系统自动关闭审查模式，已标记的保留，未标记的继续保留在列表
  - 用户可随时重新打开审查模式
  - 这样既保留了人工质量把关，又不会强制中断流程

#### 2.2 **笔记修改后重新撰写**
- **合理性**: ⭐⭐⭐⭐⭐
  - 用户修改笔记后，重新执行 Writer 是必要的
  - 保证最终综述基于修改后的笔记生成
- **技术难度**: 低
- **实现方案**:
  - v1（仅此版本）：笔记修改完毕后，提供"重新撰写综述"按钮
  - 触发后回到 `Writing` 状态，调用 Writer Agent 基于新笔记生成初稿
  - **不支持** 修改关键词后回溯到 Search 阶段（避免复杂性）
  - 若要修改搜索策略，用户可创建新会话

## 总体架构改造

### 核心概念：Session（会话）

每一次用户的"学术调研任务"作为一个 Session，在 Session 生命周期内：
- 保留完整的中间状态（Plan、Papers、Notes、Draft）
- 支持用户随时暂停、查看、编辑
- 支持多次循环执行（Re-execute）某个阶段

### 数据模型变化

#### 原 (迭代二)
```
/documents/{run_id}/
  ├── plan.md
  ├── research_notes.md
  └── final_review.md
```

#### 新 (迭代三)
```
/sessions/{session_id}/
  ├── metadata.json          # Session 基础信息（用户、主题、创建时间）
  ├── state.json             # 当前状态机（Plan|Searching|SearchComplete|ReviewingNotes|Writing|Complete）
  ├── plan/
  │   ├── initial_plan.md    # 初始规划
  │   └── confirmed_keywords.json  # 用户确认后的关键词
  ├── papers/
  │   ├── {arxiv_id}/
  │   │   ├── metadata.json  # 来源（agent_search | user_upload）
  │   │   ├── content.txt    # 论文内容/摘要
  │   │   └── {arxiv_id}.pdf
  │   └── papers_list.json   # 全部论文列表（含用户筛选状态）
  ├── notes/
  │   ├── draft_notes.md     # 所有笔记的合并版
  │   └── note_edit_history/ # 编辑历史
  ├── draft/
  │   ├── draft_v1.md        # Writer 初稿
  │   ├── user_feedback.md   # 用户反馈
  │   └── draft_v2.md        # 修改后的版本
  └── traces/                # Agent 运行轨迹（复用迭代二的记录）
```

### 状态机设计

```
[Initial]
    ↓ (用户输入主题)
[Plan]
    ↓ (Agent 生成初始规划)
[PlanConfirm] ← (用户审核、编辑关键词后确认)
    ↓
[Searching]
    ↓ (Researcher Agent 搜索论文)
[SearchComplete]
    ↓ (用户可上传补充论文、启用审查模式筛选)
[ReviewingNotes]
    ├─ (用户编辑笔记) →
    │  [Writing] → [ReviewingDraft] → (用户提供反馈) →
    │  [Writing] (最多 2 次重写)
    │     ↓
    └─ (用户确认笔记) →
    [Writing]
    ↓
[ReviewingDraft] ← (用户查看初稿、提供反馈)
    │                ↓ (反馈重写)
    └─ [Writing]    ← (支持从这里回溯重写，最多 2 次)
    │
    ↓
[Complete] ← (最终交付或用户保存为草稿)
```

**关键特性**：
- 不支持从 `ReviewingNotes` 回到 `Searching`（避免无限回溯）
- 笔记修改后，只支持重新撰写综述（v1），无需限制修改次数
- 综述反馈重写最多 2 次，防止无限修改陷阱
- 用户可随时创建新会话，若需要重新调整搜索策略

## 详细实现方案

### Phase 1: Session 管理与数据持久化

#### 1.1 后端改造（`eval_platform/backend`）

**新增模块**：`backend/session_manager.py`
```python
class SessionManager:
    def create_session(topic: str) -> Session
    def load_session(session_id: str) -> Session
    def update_session_state(session_id: str, state: str, data: dict)
    def save_papers_list(session_id: str, papers: list)
    def save_notes(session_id: str, notes_md: str)
    def save_draft(session_id: str, draft_md: str, version: int)
    def add_user_paper(session_id: str, paper_source: {title|doi|pdf})
    def delete_paper(session_id: str, paper_id: str)
    def get_session_summary(session_id: str) -> dict
```

**新增 API 端点**：
```
POST   /api/sessions/create           # 创建新会话
GET    /api/sessions/{session_id}     # 获取会话详情
PUT    /api/sessions/{session_id}/state  # 更新状态
GET    /api/sessions/{session_id}/papers # 获取论文列表
POST   /api/sessions/{session_id}/papers # 上传用户论文
DELETE /api/sessions/{session_id}/papers/{paper_id}  # 删除论文
GET    /api/sessions/{session_id}/notes   # 获取笔记草稿
PUT    /api/sessions/{session_id}/notes   # 编辑笔记
GET    /api/sessions/{session_id}/draft   # 获取综述初稿
POST   /api/sessions/{session_id}/draft/feedback  # 提交修改反馈
```

#### 1.2 Agent 改造（`agent/`）

改造 `main.py` 的 `run_agent_pipeline()`：
```python
# 原逻辑：全量同步执行
def run_agent_pipeline(user_topic: str) -> dict

# 新逻辑：支持从指定断点继续
def run_agent_pipeline(
    session_id: str,
    start_phase: str = "plan",  # 可选：plan|search|write
    user_keywords: dict = None,   # 用户确认的关键词（可选）
    known_papers: list = None,    # 已有论文列表（可选）
) -> dict
```

在各个断点后返回：
- `Plan` 完成后：返回 `keywords` + 等待用户确认
- `Search` 完成后：返回 `papers_list` + 等待用户筛选/上传
- `Notes` 完成后：返回 `notes_md` + 等待用户编辑
- `Write` 完成后：返回 `draft_md` + 等待用户反馈

## 风险评估与缓解

### 风险 1：状态管理复杂度爆炸

**表现**：支持多次回溯和重执行后，状态机非常难维护  
**影响等级**：🔴 高  
**缓解方案**：
- 严格定义状态转移图，用枚举（Enum）实现
- 编写状态机单测，每个状态转移都测试
- 不同阶段的数据隔离（`session/plan/`、`session/papers/` 等独立目录）

### 风险 2：无限修改陷阱

**表现**：用户反复修改笔记和综述，导致流程无限循环  
**影响等级**：🟡 中  
**缓解方案**：
- 笔记修改后重新撰写：最多支持 1 次（v1 only），无需限制修改次数
- 综述反馈重写：在 UI 上显示"修改次数"，超过 2 次提示用户
- 记录所有版本（`draft_v1`, `draft_v2`），允许回滚
- 如果用户需要重新调整搜索策略，建议创建新会话而不是无限回溯

### 风险 3：Session 持久化与清理

**表现**：长期运行后，Session 目录爆炸，磁盘占用飙升  
**影响等级**：🟡 中  
**缓解方案**：
- 定期清理（TTL）：超过 7 天未访问的 Session 自动清理
- 记录 Session 大小，超过阈值（如 500MB）时警告
- 定期维护任务：压缩、存档旧 Session

### 风险 4：PDF 提取失败

**表现**：用户上传的 PDF 可能是扫描件、加密或破损，无法提取  
**影响等级**：🟡 中  
**缓解方案**：
- 前端上传时显示提取预览，允许用户手动编写笔记
- 降级策略：若自动提取失败，展示"请手动输入论文信息"表单
- 记录失败原因，便于后续手工处理

### 风险 5：API 并发冲突

**表现**：用户同时编辑笔记和触发重执行，造成状态冲突  
**影响等级**：🟡 中  
**缓解方案**：
- 加乐观锁（`version` 字段）：每次更新时检查版本号
- 后端锁定：修改状态时，禁止同时进行其他操作
- 前端：状态变更时禁用按钮，防止重复提交

## 总结与建议

### 📌 关键创新点

1. **Session 会话模型** - 每个任务现在是一个持久化的会话，中间可暂停/修改/继续，无需重新启动整个流程
2. **5 层状态机设计** - 从 Plan → Searching → ReviewingNotes → Writing → Complete，支持"笔记修改→重新撰写综述"的循环，但避免无限回溯
3. **论文混合来源** - Agent 搜索结果 + 用户上传（标题/DOI/PDF）统一编辑管理，灵活应对用户的临时补充需求
4. **智能确认机制** - 论文审查模式可选启用，1 分钟无操作自动关闭，既保证质量把关，又不强制中断用户流程
5. **版本管理与回滚** - 笔记和综述都支持多版本追踪，用户可随时查看历史版本、比较改动、回滚到之前的版本

### ✅ 可行性结论

经过本次优化方案完善后，**所有提议功能都已升级为"✅ 高优先级"且技术上完全可行**：
- **关键词确认**：由"黑盒规划"改为"用户审核+确认"，避免前期偏差
- **论文管理**：从"单一 Agent 搜索"改为"混合来源+智能筛选"，满足真实场景
- **笔记编辑**：从"只读展示"改为"在线修改+版本管理"，提升调研质量
- **论文审查**：从"强制确认"改为"可选模式+自动关闭"，防止交互疲劳
- **综述重写**：从"一稿定音"改为"反馈改进+最多 2 次重写"，确保最终产物

### 📋 建议的执行路线

从**收益-风险比**和**技术依赖链**来看，建议分三波实现：

| 波次 | 功能模块 | 工作量 | 核心价值 |
|------|---------|--------|---------|
| **第一波** | Session 框架 + 关键词确认 | 12 天 | 打破"黑盒"，用户可参与关键决策 |
| **第二波** | 论文管理 + 混合来源 + 智能确认 | 11.5 天 | 支持真实调研流程（手动补充+质量把关） |
| **第三波** | 笔记编辑 + 综述重写 | 8.5 天 | 提升最终交付物质量和用户满意度 |

**总计工作量**: 31.5 人天（~5 周），安全可控

### ⚠️ 实施建议

团队在执行时：
1. **务必分波进行**：不要同时启动三波，否则会造成状态管理的混乱
2. **每波之间预留 2-3 天**：用于集成测试和问题修复
3. **持续关注"无限修改陷阱"**：虽然有防护（修改次数限制），但监控用户行为，如有卡顿立即调整
4. **准备降级方案**：PDF 提取失败、API 超时等异常要有友好的降级提示
5. **定期维护 Session**：上线前建立清理机制（TTL + 大小监控），防止磁盘爆炸

这套方案的优势在于**既保留了系统的自动化优势（Agent 搜索+撰写），又充分赋予了用户的控制权和修改权**，使得整个平台从"一次性输出工具"升级为"迭代调研助手"。
