# Academic Research Agent 需求文档

**版本**：v3.0  
**日期**：2026-06-18  
**范围**：迭代三学术调研 Agent Web 平台、CLI 入口、评估与文档支撑

---

## 1. 项目背景

Academic Research Agent 面向学术文献调研场景，目标是帮助用户从一个研究主题出发，完成关键词规划、论文检索、论文管理、深度笔记、文献分析、综述写作和后续修订。

迭代三的重点不再是单次自动生成综述，而是把系统升级为可交互、可追踪、可修订、可评估的调研助手。系统同时支持：

- **分步模式**：用户在关键节点审核和修改。
- **自动模式**：系统一键执行完整 Pipeline。

---

## 2. 用户角色

| 角色 | 需求 |
|------|------|
| 研究者 / 学生 | 快速围绕主题收集论文、形成笔记和综述 |
| 课程评审 / 助教 | 查看系统是否满足迭代三任务要求、是否可运行、是否有评估支撑 |
| 小组开发者 | 维护 Agent Pipeline、前后端功能、评估脚本和文档 |

---

## 3. 核心目标

1. **连续对话**：同一研究任务下支持多轮问答、上下文管理和多会话聊天。
2. **交互式调研**：关键词、论文、笔记、分析、综述均允许人工介入。
3. **端到端自动化**：提供自动模式，一键执行 `规划 -> 搜索 -> 笔记 -> 分析 -> 综述`。
4. **混合论文来源**：支持 Agent 搜索、arXiv/链接添加、PDF 上传等来源。
5. **可编辑中间产物**：笔记、分析卡片和综述均以 Markdown 形式编辑。
6. **可观测 Agent**：展示工具调用轨迹、错误类型、Skill 加载/回退状态。
7. **跨 Session 知识共享**：首页全局 Copilot 可跨项目问答，并支持选择知识范围。
8. **评估支撑**：保留 evaluation 目录，提供评估数据、脚本和 fallback。

---

## 4. 功能性需求

### 4.1 Session 管理

- 用户可以创建、打开、删除研究 Session。
- 每个 Session 保存主题、状态、关键词、论文、笔记、分析、综述、聊天记录和执行轨迹。
- 系统应支持状态机转移和卡住状态自动修复。
- 用户可从已有 Session 继续工作。

### 4.2 规划阶段

- 用户输入研究主题后，系统生成初始检索计划和关键词。
- 用户可以审核、编辑、保存关键词。
- 搜索阶段应使用用户确认后的关键词。

### 4.3 搜索与论文管理

- Agent 能调用 arXiv、OpenAlex、Crossref 等学术工具检索论文。
- 系统应记录搜索阶段的 ReAct 轨迹。
- 搜索结果应登记到当前 Session 的论文列表。
- 用户可以手动添加论文、上传 PDF、删除论文、修改论文状态。
- Paper Register 应保证论文登记与 PDF 下载尽量一致。

### 4.4 笔记阶段

- 用户可选择论文生成笔记。
- 系统使用 RAGNoteGenerator 基于 PDF 全文或摘要生成结构化笔记。
- 笔记应支持 Markdown 编辑和保存。
- PDF 提取失败或 Embedding 失败时，应有摘要或 BM25 回退。

### 4.5 分析阶段

- 系统应生成三类分析：文献对比、研究脉络、研究空白。
- 分析结果应保存到 `sessions/{session_id}/analysis/analysis_results.json`。
- 前端应以三张可编辑 Markdown 卡片展示分析内容。
- 自动模式中，分析阶段应在综述写作前执行。
- Writer 应读取分析结果作为额外写作上下文；没有分析结果时保持旧流程。

### 4.6 综述写作与修订

- Writer 根据笔记和分析上下文生成综述。
- 综述应保存版本，并支持 Markdown 编辑。
- 用户可以通过对话反馈触发综述修订。
- 系统应记录综述引用了哪些论文，便于论文卡片展示。

### 4.7 对话系统

- 每个 Session 支持多个聊天会话。
- 对话应保存历史，并支持上下文用量统计和压缩。
- 普通问答应优先基于当前 Session 的论文、笔记和综述。
- Agent 模式下，系统应识别用户是否在请求修订，并在必要时二次确认。

### 4.8 全局 Copilot

- 首页提供全局 Copilot。
- Copilot 能从所有真实 Session 的论文摘要、笔记、综述草稿中检索资料。
- 用户可显式选择知识检索范围。
- 未传 `session_ids` 时默认全局检索；传入时只检索选中 Session。

### 4.9 Skills 管理

- 用户可以创建、编辑、删除 search / notes / write 三类 Skill。
- 创建 Session 时可绑定 Skill。
- Skill 不可用时系统应回退默认策略。
- search / notes / write 三阶段应记录 `SKILL_STATUS` trace，说明是否加载和是否回退。

### 4.10 工具管理

- 系统提供工具注册中心。
- 用户可在前端开启/关闭工具，并持久化配置。
- 工具应标注类别和适用阶段。

---

## 5. 非功能性需求

| 类型 | 要求 |
|------|------|
| 可用性 | Web 平台可通过 `python web_app.py` 启动，默认访问 `http://127.0.0.1:8000` |
| 可维护性 | 后端路由模块化，避免所有逻辑堆在 `web_app.py` |
| 可观测性 | Agent 执行轨迹、错误类型、Skill 状态可查看 |
| 兼容性 | 分析结果或 Skill 不存在时不应中断旧流程 |
| 鲁棒性 | 外部 API 限流、PDF 解析失败、Embedding 失败时应降级 |
| 可评估性 | evaluation 目录保留评估数据、脚本和说明 |

---

## 6. 当前限制

- 生成质量依赖 LLM 的稳定性。
- 部分外部搜索 API 存在限流，部分工具默认禁用属于工程策略。
- 评估数据集仍可继续扩充，尤其是自动模式、分析注入综述、Skill trace 和跨 Session Copilot 范围选择。
- PDF 扫描件或损坏文件的解析效果有限。

---

## 7. 验收标准

- 用户能从首页创建 Session 并进入控制台。
- 分步模式可完成规划、搜索、笔记、分析、综述。
- 自动模式可一键执行完整流程。
- 分析结果可编辑，并能参与综述写作。
- 全局 Copilot 可跨 Session 问答，并支持选择范围。
- Skills 可创建、绑定，并在 trace 中展示加载/回退状态。
- `pytest -q` 能通过现有测试。
