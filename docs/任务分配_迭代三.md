## 概述与分工策略

### 三波递进开发

```
第一波（打地基）→ 第二波（建功能）→ 第三波（做交付）
```

每波之间预留 **2 天**进行集成测试与问题修复。

### 组队方式

每波任务数量为 6 个，可灵活组队：
- **单人认领**：适合独立模块（如纯前端组件、独立 API）
- **双人结对**：适合跨技术栈的任务（如一个写后端 API + 一个写前端 UI）
- **接口联调角色**：建议指定 1 人专门负责将新 API 注册到 `web_app.py`、整理接口文档、协助前后端联调，让其他组员专注业务逻辑

> ⚠️ **开发顺序建议**：每个波次内先完成后端 API（为前端提供数据源），再开发前端 UI。接口文档先行，前后端可并行。

## 第一波：核心状态管理 + 关键词确认

### 目标
搭建 Session 会话框架，让 Agent 从"一次性执行"变成"有状态的对话式执行"，并在关键词规划环节加入用户交互确认。

### 任务清单

---

#### 任务 1.1：后端 SessionManager 类

**需求**：实现 `session_manager.py`，作为整个迭代三的数据核心。

**功能要求**：
- `create_session(topic)` → 创建新 Session，生成唯一 ID，初始化文件目录
- `load_session(session_id)` → 从磁盘加载 Session 对象
- `update_session_state(session_id, new_state, data)` → 更新状态和附加数据
- `save_papers_list(session_id, papers)` → 保存论文列表
- `save_notes(session_id, notes_md)` → 保存笔记草稿
- `save_draft(session_id, draft_md, version)` → 保存综述版本

**数据结构**（文件目录结构）：

```
/sessions/{session_id}/
  ├── metadata.json          # 主题、创建时间、当前状态等
  ├── plan/
  │   └── confirmed_keywords.json
  ├── papers/
  │   ├── papers_list.json
  │   └── {arxiv_id}/
  │       ├── metadata.json
  │       └── {arxiv_id}.pdf
  ├── notes/
  │   └── draft_notes.md
  ├── draft/
  │   └── draft_v1.md
  └── traces/
      └── run_traces.json
```

**参考位置**：`backend/session_manager.py`（需新建）

#### 任务 1.2：Session 基础 API 端点

**需求**：在 `web_app.py` 中注册 Session 管理的 RESTful 接口。

**接口列表**：

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/sessions/create` | 创建新 Session |
| `GET` | `/api/sessions/{session_id}` | 获取 Session 完整状态 |
| `PUT` | `/api/sessions/{session_id}/state` | 更新 Session 状态 |
| `GET` | `/api/sessions/list` | 获取所有 Session 列表 |

**状态机枚举**：

```
planning → plan_confirmed → searching → search_complete
→ reviewing_notes → writing → reviewing_draft → complete
```

**数据模型**：

```json
{
    "session_id": "sess_20260515_abc123",
    "topic": "LLM Agent Memory",
    "status": "planning",
    "created_at": "2026-05-15T10:30:00Z",
    "updated_at": "2026-05-15T10:35:00Z"
}
```

#### 任务 1.3：Agent 断点改造

**需求**：改造 Agent 的执行流程，使其支持在关键阶段暂停并返回中间结果。

**功能要求**：
- `run_agent_pipeline()` 增加 `start_phase` 参数（可选值：`plan` / `search` / `write`）
- Plan 阶段完成后不自动进入 Search，而是返回关键词候选项，等待用户确认
- Search 阶段完成后返回论文列表，不自动进入 Writing
- 提供 Agent 断点 API 的接口文档（JSON Schema）

**状态转移流程**：

```
[planning] → (Agent 生成关键词) → 返回关键词候选项 → 等待用户确认
  → [plan_confirmed] → [searching] → (Agent 搜索论文)
  → 返回论文列表 → 等待用户筛选 → [search_complete]
  → [reviewing_notes] → [writing] → [reviewing_draft] → [complete]
```

#### 任务 2.1：论文列表 API

**需求**：实现论文的 CRUD 接口，论文数据与 Session 绑定。

**接口列表**：

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/sessions/{session_id}/papers` | 获取论文列表 |
| `DELETE` | `/api/sessions/{session_id}/papers/{paper_id}` | 删除单篇论文 |
| `POST` | `/api/sessions/{session_id}/papers/batch-delete` | 批量删除 |

**数据模型**：

```json
{
    "paper_id": "2601.05960v2",
    "title": "Distilling Feedback...",
    "source": "agent_search",
    "source_type": "arxiv",
    "status": "pending",

    "url": "https://arxiv.org/abs/2601.05960v2",
    "added_at": "2026-05-15T10:35:00Z"
}
```

**筛选支持**：
- 按来源筛选（agent_search / user_upload）
- 按状态筛选（accepted / rejected / pending）

**参考位置**：`backend/session_manager.py`（扩展现有）

---

#### 任务 2.2：用户上传论文（三种方式）

**需求**：允许用户手动补充论文到 Session 中，支持三种输入方式。

**接口**：`POST /api/sessions/{session_id}/papers`

**方式 A — 标题搜索**：
- 请求参数：`{ "type": "title", "title": "论文标题" }`
- 内部调用 `arxiv_search` 搜索匹配论文
- 确认匹配后自动下载 PDF
- 返回论文元数据和预览

**方式 B — DOI/ID**：
- 请求参数：`{ "type": "doi", "identifier": "10.xxxx/xxxxx" }` 或 `{ "type": "arxiv_id", "identifier": "2308.11432" }`
- 调用 `crossref_fetch_doi` 或 `arxiv_fetch` 获取元数据
- 有 PDF 时自动下载

**方式 C — PDF 直传**：
- 请求参数：`multipart/form-data` 文件上传
- 保存到 `sessions/{session_id}/papers/{uuid}/` 目录
- 调用 PDF 提取工具获取文本内容

**参考位置**：`backend/session_manager.py`（扩展）

---

#### 任务 2.3：PDF 提取与笔记自动生成

**需求**：上传 PDF 后自动提取内容，并调用 LLM 生成笔记草稿。

**功能要求**：
- 对上传的 PDF 调用 `ArxivPdfReaderTool` 提取文本
- 调用 LLM 生成结构化的笔记草稿（标题、作者、核心方法、关键发现）
- 返回笔记文本给前端预览
- **降级策略**：提取失败时返回友好提示，允许用户手动填写

**流程图**：

```
PDF 上传 → PDF 提取 → LLM 笔记生成 → 返回笔记预览
                         ↓ (失败)
                    "提取失败，请手动填写"
```

**参考位置**：`agent/tools/pdf_tools.py` + `agent/llms/client.py`

#### 任务 2.5：论文智能确认 + 自动关闭

**需求**：提供可选的"审查模式"，让用户对每篇论文进行快速判断，同时避免强制交互疲劳。

**功能要求**：
- 页面顶部增加"启用审查模式" toggle 开关
- 启用后，每篇论文出现三个操作按钮：
  - ✓ **接受** → 标记为 accepted
  - ✗ **拒绝** → 标记为 rejected，从笔记候选列表中移除
  - ? **稍后决定** → 标记为 pending
- **自动关闭**：页面显示 60 秒倒计时，用户 1 分钟内无操作自动关闭审查模式
  - 已标记的保留，未标记的保持 pending
- 用户可随时重新打开审查模式
- 底部统计：已接受 N 篇 / 已拒绝 N 篇 / 待定 N 篇

**UI 示意**：

```
┌────────────────────────────────────────────┐
│  论文审查模式 [🔛 已启用]                   │
│  状态: 自动关闭倒计时 0:58                 │
│  已标记：2 接受 | 0 拒绝 | 3 待定         │
├────────────────────────────────────────────┤
│  [✓ 接受] [✗ 拒绝] [? 稍后]  论文标题...   │
│  [✓ 接受] [✗ 拒绝] [? 稍后]  论文标题...   │

#### 任务 3.1：笔记 API 与版本管理

**需求**：提供笔记的读取、保存、版本管理接口。

**接口列表**：

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/sessions/{session_id}/notes` | 获取笔记 |
| `PUT` | `/api/sessions/{session_id}/notes` | 保存笔记编辑 |

**版本管理**：
- 每次保存时，在 `note_edit_history/` 目录下备份当前版本

#### 任务 3.3：Writer Agent 改造

**需求**：改造 Writer 逻辑，使其每次都能基于当前最新的笔记内容动态生成综述，而非一次性的只读流程。

**功能要求**：
- 新增 `write_with_notes(notes_content)` 函数
  - 接受当前笔记内容作为直接输入
  - 调用 LLM 生成综述初稿
  - 不依赖之前保存的文件
- 移除"只能写一次"的限制
- 保持现有章节结构（引言、方法对比、实验结果、局限性与未来方向）
- 输出保存为 `draft_v{N}.md`

**参考位置**：`agent/main.py`（改造 `compose_review_from_notes`）

---

#### 任务 3.4：状态转移与"重新撰写"流程

**需求**：用户编辑笔记后，点击"重新撰写综述"按钮，驱动 Session 进入 Writing 阶段并自动生成新初稿。

**功能要求**：
- "重新撰写综述"按钮 → 调用 API 更新 Session 状态为 `writing`
- 触发 Writer Agent 基于当前笔记生成新初稿
- 初稿保存为 `draft_v{version+1}.md`
- 完成后状态自动更新为 `reviewing_draft`
- **限制规则**：不支持从笔记编辑阶段回溯到搜索阶段（如需调整搜索策略请创建新 Session）

**状态流转**：

```
[reviewing_notes] → (编辑笔记) → [writing] → (生成初稿) → [reviewing_draft]
```

**参考位置**：`agent/main.py` + `agent/web_app.py`

---

#### 任务 3.5：最终综述预览与反馈提交

**需求**：展示综述初稿，允许用户提交修改反馈，支持重写流程。

**功能要求**：

- 第 2 次重写后提示"已达到修改上限，可手动编辑或创建新 Session"
- 每次重写生成新版本（draft_v2 → draft_v3）

**UI 示意**：

```

**参考位置**：`agent/frontend/index.html` + `app.js`

#### 任务 3.6：反馈重写逻辑与版本管理

**需求**：实现带用户反馈的综述重写引擎，支持多版本追踪。

**功能要求**：
- 新增 `write_with_feedback(notes, previous_draft, user_feedback, rewrite_count)` 函数
  - System Prompt 增加用户反馈上下文
  - 结合笔记内容和用户反馈进行重写
  - 输出保存为新版本
- **修改次数限制**：最多 2 次重写（draft_v1 → 反馈 → draft_v2 → 反馈 → draft_v3）
- 所有历史版本保留（`draft_v1.md`、`draft_v2.md`、`draft_v3.md`）
- 后续若需回滚，可直接加载旧版本

**参考位置**：`agent/main.py`（新增函数）
